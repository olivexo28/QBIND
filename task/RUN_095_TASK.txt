# RUN 095 — Binary-Path Reconfig Block Detection and Engine Epoch Transition Trigger

## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

* Runs 050–068 completed signed PQC trust-bundle lifecycle.
* Runs 069–075 completed local operator reload lifecycle.
* Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
* Run 087 added peer-driven propagation/apply safety specification.
* Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
* Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
* Run 090 updated the operator runbook for propagation-only behavior.
* Run 091 established fail-closed `CurrentEpochUnavailable` behavior for `activation_epoch` whenever no canonical runtime epoch source exists.
* Run 092 proved that the production `qbind-node` binary did not open `RocksDbConsensusStorage`, so `meta:current_epoch` was phantom on real binary runs.
* Run 093 introduced canonical production binary-path `ConsensusStorage` ownership at `<data_dir>/consensus`.
* Run 094 threaded the Run 093 `ConsensusStorage` handle into the binary consensus loop and added a fail-closed persistence hook that writes epoch advances through the existing atomic epoch persistence machinery.

Run 094 was partial-positive because the persistence hook exists and is wired, but release-binary Scenario 2 was not proven: the production binary consensus loop does not yet process committed reconfiguration blocks into `BasicHotStuffEngine::transition_to_epoch(...)`. Therefore, `engine.current_epoch()` does not advance on a real release-binary run, and the Run 094 persistence hook does not fire end-to-end on a canonical binary-path epoch transition.

Current open C4 items include:

* binary-path reconfig block detection and engine epoch transition trigger
* release-binary committed-epoch restart proof
* activation_epoch real runtime source
* snapshot epoch parity
* peer-driven live apply
* KMS/HSM custody
* in-binary/on-chain signing-key ratification
* production fast-sync / consensus-storage restore
* per-environment production trust-anchor operation
* full C4/C5 closure

Run 095 must address only the first item:

> detect canonical committed reconfig blocks on the production binary path and call the existing engine epoch-transition machinery so the Run 094 persistence hook can fire on real epoch transitions.

---

# Exact objective

Implement the smallest production-honest binary-path wiring that detects committed canonical reconfiguration blocks and triggers `BasicHotStuffEngine::transition_to_epoch(...)` or the repo’s existing equivalent epoch-transition method.

The objective is:

1. Identify the existing canonical reconfig-block representation.
2. Detect when such a block becomes committed on the binary consensus path.
3. Extract the canonical target epoch from that committed reconfig block.
4. Call the existing engine epoch-transition method.
5. Ensure Run 094 persistence then observes `engine.current_epoch()` advancing and persists the transition.
6. Ensure the persisted `reconfig_block_id` is the actual committed reconfiguration block ID.
7. Preserve Run 091/092 fail-closed trust-bundle activation behavior.

Run 095 must make engine epoch transition real on the production binary path.

Run 095 must **not** consume `current_epoch` for trust-bundle activation.

---

# Strict scope

Allowed:

## A. Reconfig-block discovery

Investigate the existing canonical representation for reconfiguration / epoch-transition blocks.

Use only existing protocol semantics.

Do not invent a new reconfiguration format unless the repo already contains no usable representation and the final verdict is partial-positive.

---

## B. Binary-path commit detection

Wire detection at the smallest safe place where the production binary path already observes committed blocks.

Expected areas may include:

* `binary_consensus_loop`
* `BasicHotStuffEngine` commit output
* existing commit notification/action surfaces
* existing block/header/body metadata
* existing reconfiguration marker structures

The detection must occur only after the block is committed under existing HotStuff commit rules.

---

## C. Engine transition trigger

When a committed canonical reconfig block is detected:

* extract the target epoch
* validate transition monotonicity using existing rules
* call `BasicHotStuffEngine::transition_to_epoch(...)` or existing equivalent
* ensure `engine.current_epoch()` advances only after a valid committed transition

---

## D. Correct reconfig_block_id

Run 094 used a defensive fallback:

```rust
engine.committed_block().unwrap_or(zero)
```

Run 095 must remove ambiguity for real transitions.

For any real epoch transition:

* the persisted `reconfig_block_id` must be the actual committed reconfiguration block ID
* zero fallback must not be used for a real transition
* if no committed reconfig block ID exists, fail closed or return partial-positive rather than silently persisting a zero marker

---

## E. Trust activation isolation

Every production `ActivationContext` must continue to use:

```rust
current_epoch: None
```

Run 095 must not feed persisted or in-memory epoch into trust-bundle activation.
