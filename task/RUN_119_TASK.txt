# Exact objective

Wire the Run 117/118 authority anti-rollback marker into the three mutating trust-bundle surfaces:

1. startup `--p2p-trust-bundle` acceptance;
2. process-start reload-apply;
3. SIGHUP live reload.

The objective is:

1. Derive `PersistentAuthorityStateRecord` only after successful ratification verification.
2. Compare the derived marker against the persisted marker before trust mutation.
3. Reject conflict, corruption, wrong-domain, or invalid marker state before mutation.
4. Persist the accepted marker atomically after the existing trust-bundle mutation commit boundary.
5. Preserve all existing trust-bundle validation checks and mutation ordering guarantees.
6. Produce release-binary evidence for first-write acceptance, same-marker idempotence, conflicting-marker rejection, and corrupt-marker fail-closed behavior if feasible.
7. Do not wire validation-only surfaces unless the three mutating surfaces are complete and the change remains clearly narrow.
8. Do not implement `--allow-authority-state-reset` unless it remains narrow and fully evidenced; otherwise defer it.
9. Do not invent a fake per-key monotonic authority sequence.
10. Do not implement signing-key rotation/revocation lifecycle.
11. Do not implement peer-driven live apply.

Run 119 should make this statement true for the three mutating surfaces:

> A ratification-enforced mutating trust-bundle path cannot silently accept a conflicting or corrupt authority marker; conflicts fail closed before trust mutation, and accepted mutations persist the authority marker atomically after the existing sequence commit boundary.

# Strict scope

Allowed:

## A. Mutating surfaces only

Wire exactly these surfaces first:

### 1. Startup trust-bundle load

The path that accepts an initial `--p2p-trust-bundle` and may persist sequence/root state or start P2P with accepted roots.

### 2. Process-start reload-apply

The Run 112/113 process-start reload-apply path.

### 3. SIGHUP live reload

The Run 114/115 long-running SIGHUP live reload path.

Do not wire validation-only surfaces in Run 119 unless the three mutating surfaces are complete, tests are green, and the validation-only change is demonstrably trivial. Prefer deferring validation-only wiring to Run 120 or Run 121.

## B. Marker derivation from verified ratification only

Use Run 118:

```text
derive_authority_state_from_ratification(...)
```

The marker must be derived only from the verified `(BundleSigningRatification, RatifiedBundleSigningKey)` pair.

Forbidden:

- deriving marker from unverified sidecar JSON;
- deriving marker from local config alone;
- deriving marker from trust-bundle sequence;
- deriving marker from activation height or activation_epoch;
- deriving marker from peer-provided data without local verified authority context.

## C. Compare-before-mutation

Use Run 118:

```text
prepare_marker_for_acceptance(...)
```

Before any trust mutation:

- load persisted marker;
- fail closed on load/corruption/malformed marker;
- validate persisted marker domain against runtime environment, chain_id, and genesis_hash;
- compare candidate marker;
- reject conflict before mutation;
- do not repair, delete, or silently overwrite corrupt markers.

## D. Persist marker after accepted mutation

Use Run 117:

```text
persist_authority_state_atomic(...)
```

Run 118 evidence recommended persisting the marker **after `commit_sequence`** so a crash window leaves the authority marker stale-by-one and safely replayable as an `Upgrade`, never ahead of the trust-bundle sequence.

Run 119 must either:

- follow this `after commit_sequence` rule; or
- document a stronger ordering with proof.

Do not claim cross-store atomicity unless the implementation actually provides it.

## E. Preserve existing ordering

### Startup

Do not weaken startup trust-bundle verification, sequence persistence, root merge, or P2P startup ordering.

### Process-start reload-apply

Preserve Run 070/073 ordering:

```text
validate → snapshot → swap → evict_sessions → commit_sequence → persist_authority_marker
```

### SIGHUP live reload

Preserve Run 074 ordering:

```text
trigger → validate → snapshot current trust → swap live trust → evict_sessions → commit_sequence → persist_authority_marker
```

The marker comparison must happen before mutation begins. The marker persistence must happen only after the existing sequence commit boundary unless a safer ordering is proven.

## F. Typed error propagation

Add or reuse precise errors for:

- marker derivation failure;
- authority marker load/corruption failure;
- wrong chain;
- wrong environment;
- wrong genesis;
- root mismatch;
- key mismatch;
- same logical root/key with different ratification digest;
- marker persist failure.

Avoid collapsing marker failures into generic reload failure when a precise reason exists.
