# Exact objective

Produce release-binary evidence proving that snapshot restore cannot silently roll back, conflict with, erase, rewrite, or repair the locally persisted authority marker.

Run 125 should prefer **no production runtime code changes**.

Allowed changes are limited to:

- release-binary evidence scripts;
- evidence-only fixture/helper generation;
- log/assertion helpers;
- documentation/evidence updates;
- narrow bug fixes if release-binary evidence exposes a real Run 124 defect.

The objective is to prove on real `target/release/qbind-node` binaries:

1. Legacy snapshot with no local authority marker restores successfully if policy allows.
2. Legacy snapshot with an existing local marker rejects; it must not silently erase local authority state.
3. Snapshot with matching authority marker restores successfully.
4. Snapshot with conflicting authority marker rejects before restore materialization.
5. Corrupt local authority marker rejects restore fail-closed.
6. Wrong-domain snapshot marker rejects if feasible.
7. Local marker bytes remain unchanged on every rejection.
8. No B3 restore audit marker is written on rejected restore.
9. Run 097 epoch/snapshot behavior remains intact.
10. No reset/recovery flag is introduced.
11. No fake authority sequence is introduced.
12. No signing-key rotation/revocation, KMS/HSM, governance, or peer-driven live apply is introduced.

# Required release-binary evidence scenarios

Produce logs, exit codes, marker files/bytes, and a summary.

## Scenario 1 — Legacy snapshot with no local marker succeeds if policy allows

Use a legacy snapshot without `authority_state` and no local marker.

Prove:

- release binary accepts restore if policy allows;
- legacy parsing compatibility is preserved;
- no synthetic authority marker is created;
- Run 097 epoch behavior remains intact.

If policy rejects legacy snapshots even with no local marker, document that honestly and cite the expected fail-closed reason.

## Scenario 2 — Legacy snapshot with existing local marker rejects

Seed local marker, then restore legacy snapshot without authority metadata.

Prove:

- release binary rejects;
- local marker bytes remain unchanged;
- no B3 restore audit marker is written;
- no restored state is accepted/materialized.

## Scenario 3 — Matching snapshot marker restores

Seed local marker and restore snapshot with matching authority metadata.

Prove:

- restore succeeds;
- marker bytes remain unchanged;
- B3 restore audit marker behavior is as expected;
- Run 097 epoch behavior remains intact.

## Scenario 4 — Conflicting snapshot marker rejects

Seed local marker and restore snapshot with conflicting authority metadata.

Prove:

- release binary rejects before restore acceptance;
- local marker bytes remain unchanged;
- no B3 restore audit marker is written;
- no restored state is accepted/materialized.

## Scenario 5 — Corrupt local marker rejects restore

Corrupt local authority marker and attempt restore.

Prove:

- release binary fails closed;
- corrupt marker is not repaired, deleted, or overwritten;
- no B3 restore audit marker is written;
- no restored state is accepted/materialized.

## Scenario 6 — Wrong-domain snapshot marker rejects, if feasible

Use wrong chain/environment/genesis marker in snapshot metadata.

Prove fail-closed behavior.

If not feasible, document as covered by Run 124 source/integration tests only.
