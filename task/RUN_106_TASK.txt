## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

- Runs 050–068 completed signed PQC trust-bundle lifecycle.
- Runs 069–075 completed local operator reload lifecycle.
- Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.

- Runs 091–099 established, wired, and partially release-binary-evidenced canonical `activation_epoch` runtime-source behavior while preserving fail-closed behavior when no committed epoch exists.
- Run 100 added the formal production trust-anchor authority and bundle-signing-key ratification model.

  - `qbind-ledger::enforce_bundle_signing_key_ratification`;

  - no network/admin/watcher/metric/dependency surface;
  - no peer-driven apply, rotation, revocation, KMS/HSM, governance, or authority anti-rollback.

- live peer-candidate validation / propagation remained unchanged;
- reload-apply remained unchanged;
- SIGHUP live reload remained unchanged;
- MainNet was not yet universally protected by default.

Run 106 must complete enforcement coverage and MainNet default strictness while preserving all mutation-ordering and non-mutation invariants.

# Exact objective

Complete ratification enforcement coverage across all trust-bundle validation surfaces and make MainNet ratification enforcement mandatory by default.

The objective is:

1. Make MainNet require bundle-signing-key ratification by default on every trust-bundle acceptance/validation surface.
2. Preserve DevNet/TestNet explicit non-production allowances only where necessary and only with clear operator visibility.
3. Wire ratification enforcement into remaining surfaces:
   - startup `--p2p-trust-bundle` default behavior;
   - `--p2p-trust-bundle-reload-check` default behavior;
   - `--p2p-trust-bundle-peer-candidate-check`;
   - live peer-candidate wire validation;
   - propagation-before-rebroadcast validation;
   - reload-apply;
   - SIGHUP live reload.
4. Ensure rejection happens before any mutation or propagation side effect.
5. Preserve every existing trust-bundle check:
   - signature verification;
   - chain_id;
   - environment;
   - sequence anti-rollback;
   - activation-height;
   - `activation_epoch`;
   - Run 065 minimum-margin;
   - revocation activation;
   - local leaf self-check;
   - local issuer-root self-check.
6. Do not implement signing-key rotation, signing-key revocation, authority anti-rollback persistence, KMS/HSM, governance, validator-set rotation, or peer-driven live apply.

Run 106 should make this statement true:

> On MainNet, every trust bundle or peer candidate signed by an unratified bundle-signing key is rejected before sequence write, root merge, live trust mutation, session eviction, sequence commit, or propagation.

# Required behavior by environment

## MainNet

MainNet must be strict by default.

Requirements:

- ratification required on all covered trust-bundle validation surfaces;
- missing ratification fails closed;
- malformed ratification fails closed;
- unknown authority root fails closed;
- transport root as authority fails closed;
- bad signature fails closed;
- wrong chain fails closed;
- wrong environment fails closed;
- incomplete authority key material fails closed;
- local configured signing key alone is not sufficient;
- no fallback roots;
- no fallback signing keys;
- no hidden bypass.

## TestNet

TestNet should prefer strict behavior.

If compatibility requires a temporary escape hatch, it must be:

- explicit;
- documented;
- disabled by default if possible;
- impossible to use on MainNet;
- recorded as a residual risk.

## DevNet

DevNet may allow explicit legacy/unratified local signing-key behavior for developer workflows, but it must be visibly non-production and impossible to confuse with MainNet.

DevNet allowances must not weaken MainNet.
