# Exact objective

Produce release-binary evidence proving that `--p2p-trust-bundle-peer-candidate-check` enforces bundle-signing-key ratification correctly under MainNet/TestNet default-strict policy and DevNet explicit opt-in policy.

Run 108 should prefer **no runtime code changes**.

Allowed code changes are limited to:

- DevNet/evidence scripts;
- test/evidence helpers;
- log assertion helpers;
- documentation/evidence updates;
- narrow bug fixes if release-binary evidence exposes a real Run 107 implementation defect.

The objective is to prove:

1. MainNet valid ratification passes peer-candidate check.
2. MainNet missing ratification rejects peer-candidate check.
3. MainNet bad ratification rejects peer-candidate check.
4. DevNet without opt-in preserves explicit legacy behavior, if that behavior is still intentionally allowed.
5. DevNet with opt-in enforces ratification.
6. Rejection remains non-mutating:
   - no sequence write;
   - no root merge;
   - no live trust mutation;
   - no session eviction;
   - no propagation;
   - no node startup;
   - no apply.
7. No trust-bundle or peer-candidate wire format changes occur.
8. No live peer-candidate dispatcher or propagation behavior changes occur.

Produce logs and exit codes for each scenario.

## Scenario 1 — MainNet valid ratification passes

Use:

- valid MainNet genesis authority with full bundle-signing authority public key material;
- valid expected genesis hash;
- peer candidate signed by a ratified bundle-signing key;
- valid ratification sidecar.

Prove:

- release binary parses genesis;

- ratification verifies;
- peer-candidate check succeeds;
- process exits successfully;
- no mutation / no node startup occurs.

## Scenario 2 — MainNet missing ratification rejects

Prove:

- peer-candidate check fails closed;
- precise missing-ratification marker appears;
- no sequence write;
- no root merge;
- no node startup;
- no propagation.

## Scenario 3 — MainNet bad ratification rejects

Use bad signature, wrong chain, or unknown authority root.

Prove:

- peer-candidate check fails closed;
- precise ratification failure marker appears;
- no mutation;
- no node startup;
- no propagation.

## Scenario 4 — DevNet without opt-in preserves explicit legacy behavior

If DevNet legacy behavior remains intentionally allowed:

- omit ratification;
- omit ratification opt-in.

Prove:

- check follows legacy path;
- output/logs make this non-production behavior explicit if the implementation supports such logging;
- no mutation;
- no node startup.

If DevNet legacy behavior no longer exists, document the new policy and adjust evidence accordingly.

## Scenario 5 — DevNet opt-in enforces ratification

Run DevNet with ratification enforcement opt-in.

Prove:

- missing/bad ratification rejects;
- valid ratification passes;
- no mutation;
- no node startup.
