Task:
Run 076. Define and implement the safest first peer/gossiped trust-bundle boundary: receive and validate peer-supplied bundle candidates, but do NOT apply them to live trust state.

Primary objective:
Add a disabled-by-default, validation-only peer/gossiped trust-bundle candidate path that can parse and validate a peer-supplied candidate using the same Run 069/073/074 security pipeline, while preventing automatic live apply, sequence burn, session eviction, or trust-state mutation.

Important framing:
This is NOT peer-driven live apply.
This is NOT automatic trust-bundle propagation.
This is NOT consensus ratification.
This is NOT activation_epoch.
This is NOT KMS/HSM.
This is NOT signing-key ratification.
This is NOT fast-sync restore.
Do not redesign KEMTLS.
Do not redesign consensus.
Do not weaken startup trust-bundle validation.
Do not weaken local operator reload-check.
Do not weaken local SIGHUP live reload.
Do not silently fall back to --p2p-trusted-root.
Do not silently fall back to DummySig, DummyKem, or DummyAead.
Do not claim full C4 or C5 closure.

Strict scope:
- Implement a peer/gossiped candidate validation boundary only.
- The path must be disabled by default.
- The path must not mutate LivePqcTrustState.
- The path must not commit sequence persistence.
- The path must not evict sessions.
- The path must not call ProductionLiveTrustApplyContext.
- The path must not accept unsigned TestNet/MainNet bundles.
- The path must not bypass chain_id / environment / activation / sequence / revocation / local self-check validation.
- The path must rate-limit or bound candidate processing to avoid DoS.
- The path must reject oversized candidates before expensive crypto.
- The path must treat all peer-supplied bytes as adversarial.
- The path must not expose private material in logs or metrics.
- Crosscheck all code against existing design/spec. If contradictions or inconsistencies are found, update docs/whitepaper/contradiction.md.
