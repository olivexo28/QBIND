Task:
Run 062. Implement and prove activation gates on revocation entries.

Primary objective:
Revocation entries in PQC trust bundles must become active only when their own activation/effective condition is satisfied. A future-dated root or leaf revocation must not reject valid certs early, while an active revocation must continue to fail closed.

Important framing:
This is a narrow revocation-activation task.
Do not implement epoch gating unless a safe pre-consensus epoch source already exists.
Do not implement KMS/HSM.
Do not implement bundle-signing-key ratification.
Do not implement per-environment minimum activation policy.
Do not implement production fast-sync restore.
Do not redesign trust bundles.
Do not redesign KEMTLS.
Do not modify consensus or timeout logic.
Do not weaken signed-bundle verification.
Do not weaken chain_id enforcement.
Do not weaken sequence anti-rollback persistence.
Do not weaken activation-height gating for bundles.
Do not weaken root/leaf revocation enforcement.
Do not silently fall back to --p2p-trusted-root.
Do not silently fall back to DummySig, DummyKem, or DummyAead.
Do not claim full C4 or C5 closure.

Strict scope:
- Locate the existing trust-bundle revocation entry model.
- If revocation entries already have effective_from, activation_height, activation_epoch, or similar fields, enforce the already-existing field.
- If only effective_from exists and it is timestamp-like, do not invent unsafe clock semantics unless a safe source already exists.
- Prefer height-based gating if activation_height exists and the Run 057 current_height source can be reused.
- Immediate revocations with no activation/effective gate must preserve existing behavior.
- Future revocations must not be active early.
- Active/satisfied revocations must fail closed exactly as Run 050/052/061 require.
- Revocation activation must be evaluated before constructing:
  - revoked root set
  - revoked leaf fingerprint set
  - local leaf self-check
  - P2P handshake revocation context
- Preserve Run 050–061 behavior.
- Crosscheck all code against existing design/spec. If contradictions or inconsistencies are found, update docs/whitepaper/contradiction.md.

3. Define revocation activation semantics
   Required policy:
   - revocation entry without activation/effective gate:
     - active immediately, preserving Run 050/052 behavior
   - revocation entry with activation_height H:
     - active iff current_height >= H
     - inactive iff current_height < H
   - inactive revocation entries:
     - must not appear in active revoked root set
     - must not appear in active revoked leaf fingerprint set
     - must not cause local leaf startup self-check failure
     - must not cause peer handshake revoked failure
   - active revocation entries:
     - root revocation excludes/rejects root as before
     - leaf revocation rejects local/peer leaf as before
   - activation fields must be signature-covered and fingerprint-covered.
   - tampering revocation activation fields after signing must fail signature verification.
   - invalid activation field must fail closed.

   - when height later reaches the revocation activation height, a restart/load should activate the revocation without requiring a new bundle.
