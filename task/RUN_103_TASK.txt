# RUN 103 — Minimal Bundle-Signing-Key Ratification Verifier

## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

- Runs 050–068 completed signed PQC trust-bundle lifecycle.
- Runs 069–075 completed local operator reload lifecycle.
- Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
- Run 087 added peer-driven propagation/apply safety specification.
- Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
- Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
- Run 090 updated the operator runbook for propagation-only behavior.
- Runs 091–099 established, wired, and partially release-binary-evidenced canonical `activation_epoch` runtime-source behavior while preserving fail-closed behavior when no committed epoch exists.
- Run 100 added the formal production trust-anchor authority and bundle-signing-key ratification model:
  - production authority must be genesis-bound;
  - production static source-code anchors are rejected;
  - MainNet local/dev shortcuts are rejected;
  - bundle-signing-key ratification remains future implementation;
  - peer-driven live apply remains forbidden until authority ratification, anti-rollback, custody, emergency controls, and release-binary evidence exist.
- Run 101 added additive genesis authority fields and canonical genesis verification API:
  - `GenesisConfig.authority: Option<GenesisAuthorityConfig>`;
  - structurally separate PQC transport roots and bundle-signing authority roots;
  - canonical genesis hash with domain separation;
  - environment-specific boot verification API;
  - no source-code MainNet root anchors;
  - no transport-root reuse as bundle-signing authority.
- Run 102 wired canonical genesis verification into the release binary startup path and fixed `--print-genesis-hash`:
  - release binary now verifies canonical genesis authority before restore / metrics / network / consensus startup;
  - `--print-genesis-hash` prints the canonical Run 101 parsed-genesis hash, not raw file-byte hash semantics;
  - MainNet fails closed on missing/malformed/mismatched expected hash and missing/malformed authority;
  - DevNet embedded-genesis path remains explicit non-production behavior;
  - the minimal ratification verifier skeleton was intentionally deferred to Run 103.

Run 103 is the next authority-layer implementation step.

The goal is to make this statement true at the verifier level:

> A bundle-signing key is not merely locally configured; it must be ratified by a genesis-bound bundle-signing authority root.

Run 103 must implement only the minimal bundle-signing-key ratification verifier. It must not implement rotation, revocation lifecycle, anti-rollback persistence, KMS/HSM custody, governance, peer-driven live apply, or trust-bundle wire-format changes.

---

# Exact objective

Implement a minimal PQC-only bundle-signing-key ratification verifier that validates a bundle-signing key authorization object against genesis-bound `bundle_signing_authority_roots` from Run 101 / Run 102.

The objective is:

1. Define a ratification object/schema for authorizing a bundle-signing public key.
2. Bind the ratification object to:
   - chain_id;
   - environment;
   - genesis hash or genesis authority context;
   - authority root fingerprint;
   - bundle-signing key fingerprint;
   - signature suite;
   - domain-separated signing preimage.
3. Verify that the authority root used for ratification exists in genesis-bound `bundle_signing_authority_roots`.
4. Verify that PQC transport roots cannot authorize bundle-signing keys.
5. Verify the ratification signature using existing PQC signature suite abstractions.
6. Expose a narrow verifier API with typed accept/reject reasons.
7. Add tests proving wrong-chain, wrong-environment, wrong-root, malformed, unsupported-suite, and bad-signature cases fail closed.
8. Do not yet require all trust-bundle runtime paths to consume ratification unless the repo already has a clean non-broad hook.

Run 103 should establish the verifier primitive. Consumption/enforcement across all trust-bundle loading surfaces may be deferred to Run 104 if wiring it everywhere would broaden scope.

# Strict scope

Allowed:

## A. Ratification object type

Add a narrow object for bundle-signing-key ratification.

Expected conceptual fields:

```text
version
chain_id
environment
genesis_hash or authority_context_hash
authority_root_fingerprint
bundle_signing_public_key_fingerprint
bundle_signing_public_key or public-key reference
signature_suite_id
activation_epoch or activation_height, only if already appropriate
validity window, only if simple and clearly specified
signature
```

Use repo-appropriate names and types.

The object must be canonicalized before signing/verifying.

## B. Domain-separated canonical signing preimage

Add a canonical signing preimage helper.

Use a project-consistent domain separator, for example:

```text
QBIND:BUNDLE-SIGNING-RATIFICATION:v1
```

If the repo already has naming conventions for domain tags, follow those.

Requirements:

- deterministic;
- chain_id-bound;
- environment-bound;
- genesis/authority-context-bound;
- authority-root-bound;
- bundle-signing-key-bound;
- suite-bound;
- no JSON map-order ambiguity;
- no raw ad hoc string concatenation if canonical encoding exists.

## C. Authority root lookup

Verifier must accept only roots from genesis-bound `bundle_signing_authority_roots`.

It must reject:

- missing authority root;
- unknown authority root fingerprint;
- transport root fingerprint used as bundle-signing authority;
- wrong-chain authority;
- wrong-environment authority;
- malformed root;
- duplicate/ambiguous authority root if repo policy rejects duplicates.

## D. PQC signature verification

Use existing PQC signature suite abstractions.

Requirements:

- no classical signatures;
- no dummy signature suite;
- no test-only verifier in production path;
- suite ID must be explicit;
- unsupported suite must fail closed;
- bad signature must fail closed.

If the authority root public key material currently lacks enough structure to verify a signature cleanly, do not fake verification. Return partial-positive and document the missing authority-key material boundary.

## E. Narrow verifier API

Expose a narrow API such as:

```rust
verify_bundle_signing_key_ratification(...)
```

or repo-style equivalent.

The API must return typed success/failure, not boolean-only results.

Expected failure classes:

- malformed object;
- unsupported version;
- chain mismatch;
- environment mismatch;
- genesis/authority-context mismatch;
- unknown authority root;
- transport root not allowed;
- bundle-signing key fingerprint mismatch;
- unsupported suite;
- bad signature;
- expired/not-yet-active, only if validity windows are included.

## F. Optional limited integration point

If clean and narrow, add a validation-only integration point that can verify a ratification object file independently of trust-bundle apply.

Possible acceptable surfaces:

- unit/integration tests only;
- hidden local check command if repo style supports it;
- library helper consumed by future Run 104.

Do not wire into all trust-bundle acceptance paths if doing so would require broad redesign.
