# Exact objective

Implement the ratification v2 schema/types, deterministic canonical preimage, domain-separated digest, and verifier test surface exactly according to the Run 129 spec.

The objective is:

1. Add ratification v2 data types without breaking v1.
2. Add deterministic v2 canonical preimage using domain tag `QBIND:BUNDLE-SIGNING-RATIFICATION:v2`.
3. Add v2 canonical digest.
4. Add v2 verifier logic and typed failure reasons.
5. Verify ML-DSA-44 signatures using the existing crypto stack only.
6. Enforce v2 domain binding: environment, chain_id, genesis_hash, authority root fingerprint, target key fingerprint, lifecycle action, and `authority_domain_sequence`.
7. Enforce lifecycle-action-specific fields:
   - `ratify` must bind target key and sequence;
   - `rotate` must bind previous key and previous ratification digest;
   - `revoke` must bind revoked key/reason/scope as specified.
8. Preserve v1 behavior unchanged.
9. Do not wire v2 into startup/reload/SIGHUP/validation-only surfaces yet.
10. Do not alter authority marker persistence behavior yet.
11. Do not implement signing-key rotation/revocation lifecycle yet.

Run 130 should make this statement true:

> QBIND has a tested ratification v2 object, canonical preimage/digest, and verifier primitive with typed fail-closed behavior, but no production enforcement path consumes v2 yet.

## C. v2 canonical preimage and digest

Implement deterministic length-prefixed encoding.

Requirements:

- domain tag distinct from v1: `QBIND:BUNDLE-SIGNING-RATIFICATION:v2`;
- no JSON ordering ambiguity;
- includes all security-relevant fields;
- includes lifecycle action;
- includes `authority_domain_sequence`;
- includes previous-key / previous-digest linkage for rotation;
- includes revoke reason/scope for revocation if present;
- changes to any security field change digest;
- deterministic across platforms.

## D. v2 verifier primitive

Add a verifier function for v2, for example:

```text
verify_bundle_signing_key_ratification_v2(...)
```

or repo-equivalent.

It must:

- use existing ML-DSA-44 verifier stack only;
- only accept roots from `bundle_signing_authority_roots`;
- reject transport roots;
- verify authority root key material from genesis-bound full public key material;
- check authority root fingerprint matches;
- check suite IDs;
- check chain_id/environment/genesis_hash;
- check target signing key fingerprint/public key binding;
- check lifecycle-action-specific required fields;
- return typed failures, not boolean-only failure.

## E. Compatibility with v1

Preserve existing v1 verifier behavior bit-for-bit except for additive code.

Do not make v1 accept v2 objects.
Do not make v2 accept v1 objects.
Do not wire v1/v2 compatibility policy into production surfaces yet.

Run 130 may add pure comparison/helper tests for v1/v2 downgrade semantics if they are strictly local and do not change production enforcement.
