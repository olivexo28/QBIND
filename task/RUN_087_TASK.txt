Required design document:
Create:
- docs/protocol/QBIND_PEER_TRUST_BUNDLE_PROPAGATION_SAFETY.md

The spec must include:

1. Scope and non-goals
   Must explicitly state:
   - current peer-candidate 0x05 is validation-only.
   - peer-driven live apply is not implemented today.
   - propagation is not implemented today.
   - this document defines gates for future work only.
   - no automatic trust-bundle synchronization is allowed without satisfying the gates.

2. Threat model
   Include:
   - malicious peers sending malformed bundles
   - oversized candidates / DoS
   - replayed old bundles
   - validly signed but rollback/equivocation bundles
   - wrong-chain / wrong-environment candidates
   - activation-height manipulation
   - premature revocation activation
   - malicious propagation flood
   - compromised bundle-signing key
   - split-brain trust-bundle distribution
   - peer attempting to trigger sequence burn
   - peer attempting to trigger session eviction
   - peer attempting to trigger local live apply
   - peer attempting to make MainNet behave like DevNet
   - peer attempting to bypass operator approval

3. Trust and authority model
   Define:
   - bundle-signing key authority
   - local operator authority
   - peer authority: advisory only unless future ratification mechanism exists
   - chain_id binding
   - environment binding
   - sequence monotonicity
   - activation-height / minimum-margin policy
   - revocation semantics
   - local leaf and issuer-root self-checks

4. Candidate lifecycle state machine
   Define states:
   - Received
   - Decoded
   - ValidationRejected
   - ValidationAccepted
   - ObservedOnly
   - OperatorSelected
   - LocalReloadPreflighted
   - LocalApplyPending
   - AppliedByLocalOperator
   - RejectedAfterPreflight
   - Expired / superseded

   Must state:
   - peer-candidate receive may only reach ObservedOnly today.
   - only local operator action may move a candidate toward apply today.
   - future peer-driven apply requires a separate run and additional authority.

5. Future propagation gates
   Define minimum gates before any propagation/rebroadcast:
   - bounded payload
   - validation before rebroadcast
   - duplicate suppression
   - rate limiting
   - no rebroadcast loops
   - no apply side effect
   - no sequence commit
   - no session eviction
   - clear metrics
   - per-peer abuse handling
   - no private material in logs

6. Future peer-driven apply gates
   Define minimum gates before any peer-triggered apply:
   - signed-bundle verification
   - chain_id / environment enforcement
   - sequence monotonicity
   - activation-height and minimum-margin enforcement
   - revocation activation filtering
   - local leaf self-check
   - local issuer-root self-check
   - local policy approval
   - bundle-signing-key ratification or equivalent authority
   - operator override / emergency controls
   - rollback handling
   - session eviction policy
   - sequence commit only after successful swap + eviction
   - fail-closed behavior
   - audit/evidence capture

7. Explicit unsafe designs to reject
   Must reject:
   - peer sends bundle and receiver immediately applies it
   - peer candidate burns sequence before local apply
   - receiver rebroadcasts unvalidated candidate
   - receiver applies unsigned bundle
   - receiver accepts wrong-chain/wrong-env bundle
   - receiver bypasses activation margins
   - receiver keeps sessions alive under revoked roots after apply
   - receiver falls back to --p2p-trusted-root
   - receiver uses DummySig / DummyKem / DummyAead
   - receiver treats 0x05 as consensus ratification
   - receiver treats majority gossip as authority without ratification

8. Metrics and evidence expectations
   Define expected counters/logs for future propagation/apply:
   - received
   - validated
   - rejected
   - duplicate
   - rate-limited
   - propagated if ever implemented
   - apply-attempt
   - apply-success
   - apply-failure
   - sequence-commit
   - session-eviction
   - rollback
   - clear split between validation-only and apply metrics

9. Open questions / required future runs
   Must list:
   - activation_epoch runtime source
   - KMS/HSM custody
   - in-binary / on-chain signing-key ratification
   - peer-driven apply policy
   - propagation topology
   - per-environment production trust-anchor operation
   - fast-sync / restore parity
   - MainNet operational playbook for trust-bundle propagation
