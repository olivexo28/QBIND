Verifier design requirements:

1. Default Disabled / fail-closed

The production on-chain governance proof verifier must default to Disabled or equivalent.

Disabled must fail closed before any proof parsing, root verification, inclusion verification, or lifecycle authorization.

2. Preserve existing fixture boundary

Existing fixture OnChainGovernance behavior must remain DevNet/TestNet evidence-only.

Fixture proof must be rejected for MainNet.

Fixture proof must not satisfy production governance authority.

Existing `AllowFixtureSourceTest` behavior must not become a production verifier selector.

3. Production proof class

Add or activate a production proof class only when verification is real.

A production proof may be accepted at source/test level only if all represented proof material is verified against explicit trusted inputs.

At minimum, the verifier should bind and check:

* environment;
* chain_id;
* genesis hash;
* authority root fingerprint and suite;
* governance domain id;
* governance epoch;
* proposal id;
* proposal digest;
* proposal outcome;
* lifecycle action;
* active / new / revoked bundle-signing key fingerprints, where represented;
* authority-domain sequence;
* candidate v2 digest;
* decision id / replay nonce;
* freshness window or explicit governance height/epoch bounds;
* quorum / threshold, if represented;
* proof suite id;
* trusted governance checkpoint/root;
* receipt root / event root / state root, if represented;
* event / decision inclusion proof path, if represented;
* proof bytes digest;
* domain separation tag.

Wrong value for any bound field must fail closed with a typed error.

4. Canonical bytes

All production proof commitments must use deterministic canonical bytes.

Do not use Rust `Debug` formatting as canonical proof bytes.

Do not use wall-clock time as an authorizing input.

Freshness must be explicit: governance height, governance epoch, authority-domain sequence, decision id, replay nonce, or typed validity window.

5. Inclusion / commitment verification

The verifier must perform real verification over the represented production proof structure.

Acceptable source/test production verifier models include:

* Merkle inclusion proof from decision/event commitment to a trusted governance root;
* receipt/event root inclusion proof tied to a trusted governance checkpoint;
* typed decision commitment verified against a trusted root with deterministic proof path;
* threshold/quorum proof over deterministic signer set if the existing proof format represents that layer.

Do not simply compare a fixture hash unless it is explicitly a real deterministic proof root and path verification algorithm.

Do not silently accept proof bytes because they are non-empty.

Do not classify an opaque blob as production verified.

6. Trusted root / checkpoint handling

Trusted governance root/checkpoint input must be explicit.

Missing trusted root must fail closed.

Wrong root must fail closed.

Untrusted root supplied by the proof itself must not be accepted as authority.

The proof cannot self-authorize its root.

7. MainNet policy

MainNet must remain refused unless production authority criteria are satisfied.

For Run 299 source/test, MainNet should remain fail-closed even for otherwise valid synthetic production-style proofs unless there is an explicit test-only MainNet source/test policy that still documents no MainNet runtime enablement and no MainNet readiness.

Fixture, local operator, RemoteSigner-only, KMS/HSM custody alone, custody attestation alone, and peer majority must not satisfy MainNet on-chain governance proof verification.

A valid on-chain governance proof alone must not bypass missing governance execution engine, validator-set rotation, production custody ceremonies, or full MainNet evidence gates.
