Task:
Run 055. Implement trust-bundle sequence-number monotonicity persistence across restarts.

Primary objective:
Prevent rollback to an older signed trust bundle after a node has already accepted a newer trust bundle for the same chain/environment trust domain.

Important framing:
This is an anti-rollback persistence task.
Do not implement activation epoch/height gating in this run.
Do not implement CA/rotation playbook in this run.
Do not implement MainNet live smoke in this run unless it is trivial and does not distract.
Do not redesign trust bundles.
Do not redesign KEMTLS.
Do not modify consensus or timeout logic.
Do not weaken signed-bundle verification.
Do not silently fall back to --p2p-trusted-root.
Do not silently fall back to DummySig, DummyKem, or DummyAead.
Do not claim full C4 or C5 closure.

Strict scope:
- Persist the highest accepted trust-bundle sequence number for the relevant trust domain.
- Reject rollback to a lower sequence after restart.
- Define and enforce equal-sequence behavior explicitly.
- Fail closed on corrupted persistence state.
- Preserve Run 050/051/052/053/054 behavior.
- Keep DevNet compatibility where safe, but do not let DevNet convenience weaken TestNet/MainNet semantics.
- Crosscheck all code against existing design/spec. If contradictions or inconsistencies are found, update docs/whitepaper/contradiction.md.

3. Define sequence semantics
   Required policy:
   - If no prior sequence exists for this trust domain:
     - accept the loaded signed/valid bundle and persist its sequence.
   - If new sequence > stored highest:
     - accept and update stored highest.
   - If new sequence < stored highest:
     - reject fail-closed as rollback.
   - Equal sequence:
     - choose and document deterministic behavior.
     - Recommended safest behavior:
       - allow equal sequence only if canonical bundle fingerprint matches the previously accepted fingerprint for that sequence;
       - reject equal sequence with different fingerprint as equivocation/replay ambiguity.
     - If this is too large, reject equal sequence always except for first load in the same process. But document operator impact.
   - Sequence 0:
     - either allow only for DevNet or reject if production policy requires positive sequence.
     - Do not invent silent special cases.
   - Missing sequence:
     - if schema currently requires it, keep required.
     - if optional, define fail-closed for TestNet/MainNet and DevNet compatibility policy.

   Preferred:

5. Define failure policy
   Required:
   - corrupted sequence file causes startup fail-closed.
   - wrong environment in sequence file causes fail-closed if same path/domain is used.
   - wrong chain_id in sequence file causes fail-closed if same path/domain is used.
   - lower sequence bundle causes fail-closed.
   - equal sequence different fingerprint causes fail-closed if adopting recommended policy.
   - persistence write failure causes fail-closed; do not accept bundle if state cannot be recorded.
   - do not silently delete corrupt state.
   - do not silently reset highest sequence.
