# Required release-binary evidence scenarios

Produce logs, metrics, exit codes, and a machine-readable summary if possible.

## Scenario 1 — Valid ratified candidate validates and may propagate

Use live N=3 release binaries.

Enable propagation if using the Run 088 propagation path.

Prove:

- V1 receives candidate from V0;
- V1 ratification-validates candidate;
- V1 rebroadcasts to V2 if propagation enabled;
- V1 excludes source peer V0 from rebroadcast;
- V2 receives candidate;
- no apply occurs;
- no sequence write;
- no live trust mutation;
- no session eviction.

## Scenario 2 — Missing ratification rejects and does not propagate

Send candidate without required ratification.

Prove:

- V1 receives candidate;
- V1 rejects at ratification gate;
- V1 does not rebroadcast;
- V2 does not receive candidate;
- no apply;
- no sequence write;
- no live trust mutation;
- no session eviction.

## Scenario 3 — Bad ratification rejects and does not propagate

Send candidate with bad signature / wrong ratification.

Prove same negative invariants as Scenario 2.

## Scenario 4 — Duplicate/seen-cache cannot convert rejection into acceptance

Send invalid/unratified candidate repeatedly.

Prove:

- duplicate handling does not create rebroadcast;
- seen-cache does not mark rejected candidate as accepted;
- no propagation.

## Scenario 5 — DevNet policy behavior

If using DevNet topology:

- prove no-opt-in behavior if legacy behavior is preserved;
- prove opt-in enforcement if ratification opt-in is enabled.

If using MainNet/TestNet topology, document why DevNet policy is covered by tests rather than release-binary run.
