# Exact objective

Design the ratification v2 schema and comparison model for per-key monotonic bundle-signing authority progression.

The objective is to define, without implementing the full rotation/revocation lifecycle yet:

1. The canonical ratification v2 object format.
2. The per-key monotonic field or fields.
3. The domain-separated signing preimage.
4. The authority-state marker changes needed to track v2 monotonic progress.
5. The backward compatibility policy for v1 ratifications.
6. The migration path from v1 to v2.
7. The comparison rules for same key, rotated key, revoked key, and conflicting ratification states.
8. The fail-closed behavior for downgrade, replay, forked sequence, wrong-domain, or ambiguous v1/v2 combinations.
9. The future implementation staging for verifier support, marker support, evidence, then rotation/revocation lifecycle.

Run 129 should make this statement true:

> QBIND has a formal ratification v2 schema that supplies the missing per-key monotonic authority field required for safe signing-key rotation and revocation, while preserving v1 safety and preventing downgrade/replay ambiguity.

# Strict scope

Allowed:

## A. Ratification v2 schema specification

Define a new v2 ratification object model.

Expected conceptual fields:

```text
schema_version = 2
environment
chain_id
genesis_hash
authority_policy_version
authority_root_fingerprint
authority_root_suite_id
ratified_bundle_signing_key_fingerprint
ratified_bundle_signing_key_suite_id
ratified_bundle_signing_public_key_hash_or_public_key_ref
per_key_authority_sequence
key_lifecycle_action
previous_key_fingerprint_if_rotation
previous_ratification_digest_if_rotation
valid_from_epoch_or_height_if already canonical
valid_until_epoch_or_height_if used
revocation_reason_if action = revoke
capabilities / scope
```

Use repo-appropriate names. Do not invent runtime semantics that are not yet canonical.

## B. Monotonic field design

Define the exact monotonic field needed for safe authority progression.

The design must answer:

- Is monotonicity global per authority root, per ratified key, or per authority domain?
- What object owns the monotonic counter?
- What prevents two conflicting ratifications at the same sequence?
- How does a node compare v2 ratification against local marker state?
- How is v1 handled when local state has already seen v2?
- How does reset/recovery interact with the monotonic field?

Prefer a clear per-authority-domain sequence with explicit key lifecycle action if safer than a narrow per-key-only sequence. If choosing per-key, justify how rotation between keys is ordered.

## C. Domain-separated canonical preimage

Define the v2 signing preimage.

Requirements:

- deterministic length-prefixed encoding;
- domain tag distinct from v1, e.g. `QBIND:BUNDLE-SIGNING-RATIFICATION:v2`;
- includes all security-relevant fields;
- no JSON ambiguity;
- chain/environment/genesis binding;
- authority root binding;
- target key binding;
- monotonic sequence binding;
- lifecycle action binding.

## D. Authority marker v2 extension

Define how `PersistentAuthorityStateRecord` must evolve.

Expected additions may include:

```text
authority_schema_version
latest_authority_sequence
key_lifecycle_action
active_key_fingerprint
previous_key_fingerprint
latest_ratification_digest
revoked_key_set_digest or future revocation accumulator
```

Do not implement it in Run 129 unless explicitly scoped later. This run is spec-first.

## E. Compatibility and migration policy

Define behavior for:

- v1 ratification on a node with no marker;
- v1 ratification on a node with v1 marker;
- v2 ratification on a node with v1 marker;
- v1 ratification after v2 marker exists;
- v2 ratification with lower sequence;
- v2 ratification with same sequence and same digest;
- v2 ratification with same sequence and different digest;
- v2 ratification with higher sequence;
- migration from v1 to v2 without local reset;
- migration after reset/recovery.

At minimum, v1 after v2 must fail closed unless a very narrow, explicit compatibility rule is proven safe.

Likely:

## 3. Rotation/revocation threat model

Document threats:

- replay older ratification for old signing key;
- equivocation with same sequence different key;
- key rollback after rotation;
- stale sidecar reuse;
- cross-environment replay;
- cross-chain replay;
- reset/recovery downgrade;
- v1 downgrade after v2 activation;
- compromised authority root signing conflicting sequences;
- node partition seeing different monotonic histories.

## 4. Monotonic design alternatives

Compare at least two options:

### Option 1 — per-key sequence

Each ratified key has its own sequence.

Potential issue: does not naturally order rotation between different keys.

### Option 2 — per-authority-domain sequence

Each authority domain has one monotonically increasing sequence across lifecycle actions.

Potential benefit: orders add/rotate/revoke actions globally inside the authority domain.

### Option 3 — epoch-bound governance sequence

Sequence tied to canonical chain epoch/governance artifact.

Potential benefit: stronger ordering, but likely requires later governance integration.

Recommend one option with explicit rationale.

## 5. Activation and downgrade policy

Define how v2 becomes accepted:

- by genesis policy version?
- by authority policy version in genesis?
- by future governance activation?
- by binary config for DevNet/TestNet only?

Define how v1 is treated after v2 activation.
