# Exact objective

Write the formal specification for authority-state reset / recovery after fail-closed anti-rollback events.

The objective is to define, without implementing a CLI bypass yet:

1. When authority-state reset/recovery is allowed.
2. When it is forbidden.
3. Which environments may support it.
4. What operator ceremony is required.
5. What evidence/audit artifacts must be produced.
6. What must be verified before reset.
7. What must be preserved before reset.
8. What marker bytes and snapshot metadata must be archived.
9. How reset interacts with genesis-bound authority, ratification, snapshot restore, and Run 055 sequence anti-rollback.
10. Why reset must not become a MainNet local-config bypass.
11. What implementation boundaries a later run must obey.

Run 126 should make this statement true:

> QBIND has a formal, security-reviewed authority-state reset/recovery procedure specification that cannot be confused with ordinary startup, restore, reload, or peer-driven behavior, and which preserves the anti-rollback security model by default.

# Required investigation

Before writing the final spec, investigate and document:

## 1. Existing authority marker failure modes

Review all fail-closed states from Runs 117–125:

- corrupt local marker;
- conflicting local marker;
- wrong-domain marker;
- legacy snapshot with local marker;
- no-genesis-context restore with local marker;
- marker persist failure after commit;
- validation-only marker conflict;
- live `0x05` marker conflict.

Classify each as:

- operator-recoverable;
- environment-reset-only;
- MainNet governance-required;
- forbidden local recovery;
- evidence-only / investigation-required.

## 2. Environment policy

Define reset policy separately for:

### DevNet

May allow explicit reset for test automation and local recovery, but must require explicit command and audit output.

### TestNet

May allow controlled reset under explicit operator ceremony, but should require stronger proof and produce audit artifacts.

### MainNet

Local reset should be disallowed by default.

Any MainNet recovery should require a future governance/ratification procedure or offline signed recovery artifact, not local config alone.

Do not implement this in Run 126; specify it.

## 3. Operator artifact requirements

Define what must be archived before any future reset:

- old `pqc_authority_state.json` bytes and hash;
- candidate new marker bytes and hash;
- genesis file hash;
- expected genesis hash;
- chain_id;
- environment;
- snapshot metadata if involved;
- ratification sidecar;
- trust bundle fingerprint;
- timestamp/operator note;
- binary sha256/build-id if used.

## 4. Reset safety invariants

Define mandatory invariants:

- reset must never run implicitly;
- reset must never run during normal startup;
- reset must never be triggered by peer input;
- reset must never persist from validation-only surfaces;
- reset must never synthesize marker from legacy snapshot;
- reset must never bypass ratification verification;
- reset must never allow transport root authority;
- reset must never allow local config alone as MainNet authority;
- reset must always produce an audit record;
- reset must be irreversible without another explicit audited ceremony.

## 5. Future CLI / command design

Define a future command shape, but do not implement it.

Example conceptual shape:

```text
qbind-node authority-state-reset \
  --data-dir <PATH> \
  --genesis-path <PATH> \
  --expected-genesis-hash <HASH> \
  --p2p-trust-bundle <BUNDLE> \
  --p2p-trust-bundle-ratification <RATIFICATION> \
  --output-audit <PATH> \
  --environment <devnet|testnet|mainnet>
```

The exact names should follow repo conventions in a future implementation run.

The spec must state MainNet behavior separately and should default to refusal unless a future governance-approved artifact exists.

# Required design content

## 1. Threat model

Include at least:

- malicious operator rollback;
- accidental data-dir restore;
- stale snapshot restore;
- stale ratification sidecar reuse;
- conflicting marker injection;
- corrupt marker recovery temptation;
- bypass via DevNet flags leaking into MainNet;
- peer-triggered reset attempt;
- local config-only authority escalation.

## 2. Recovery classification table

Create a table with columns:

```text
Failure condition | DevNet | TestNet | MainNet | Required artifacts | Future implementation behavior
```

## 3. Reset ceremony

Define the exact staged operator process:

1. stop node;
2. archive current data dir / marker / snapshot metadata;
3. compute hashes;
4. verify genesis and environment;
5. verify target ratification under genesis authority;
6. compare old/new markers;
7. produce audit plan;
8. require explicit operator confirmation;
9. write new marker only via future reset command;
10. produce final audit record;
11. restart node;
12. verify marker and trust state.

## 4. Refusal cases

The spec must require future implementation to refuse:

- missing genesis hash;
- wrong genesis hash;
- wrong chain_id;
- wrong environment;
- malformed authority root;
- transport root attempting to authorize bundle signing key;
- missing ratification;
- bad ratification;
- local config-only MainNet reset;
- peer-provided reset request;
- reset while node is running, unless a future design safely supports offline-only lock;
- missing audit output path;
- attempt to erase marker without replacement on MainNet/TestNet.

## 5. Audit record schema

Define conceptual fields for a future audit record, such as:

```text
record_version
action = authority_state_reset
environment
chain_id
genesis_hash
old_marker_hash
old_marker_record
new_marker_hash
new_marker_record
ratification_hash
trust_bundle_fingerprint
snapshot_metadata_hash_if_any
operator_note_hash
binary_sha256
binary_build_id
timestamp
result
```

No implementation is required in Run 126.

## 6. Interaction with future ratification v2 monotonic schema

State clearly:

- current marker can detect conflict/wrong-domain/corruption but does not prove per-key monotonic authority progression;
- future ratification v2 must add canonical per-key monotonic field;
- reset/recovery must not be used as a substitute for rotation/revocation lifecycle;
- any future reset after ratification v2 must respect monotonic authority sequence.
