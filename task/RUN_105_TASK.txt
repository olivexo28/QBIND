## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

- Runs 069–075 completed local operator reload lifecycle.
- Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
- Run 087 added peer-driven propagation/apply safety specification.
- Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
- Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
- Run 090 updated the operator runbook for propagation-only behavior.
- Runs 091–099 established, wired, and partially release-binary-evidenced canonical `activation_epoch` runtime-source behavior while preserving fail-closed behavior when no committed epoch exists.
- Run 100 added the formal production trust-anchor authority and bundle-signing-key ratification model.
- Run 101 added additive genesis authority fields and canonical genesis verification API.
- Run 102 wired canonical genesis verification into the release binary startup path and fixed `--print-genesis-hash`.
- Run 103 implemented the minimal bundle-signing-key ratification verifier primitive:
  - `BundleSigningRatification` schema;
  - deterministic length-prefixed canonical preimage;
  - domain separator `QBIND:BUNDLE-SIGNING-RATIFICATION:v1`;
  - typed `RatificationFailure` reasons;
  - PQC verification through existing `MlDsa44SignatureSuite` / suite `100`;
  - authority lookup restricted to `bundle_signing_authority_roots`;
  - transport roots rejected as bundle-signing authority;

  - `GenesisAuthorityRoot::public_key_hex`;

  - full key material is included in canonical genesis hash;
  - MainNet rejects fingerprint-only bundle-signing authority roots;
  - Run 103 verifier can verify ratification signatures using genesis-bound authority public key material;
  - `AuthorityKeyMaterialUnavailable` is preserved for incomplete/legacy roots;
  - malformed/tampered material fails closed with `AuthorityKeyMaterialMalformed`;
  - no trust-bundle apply-path enforcement was added.

Run 104 was positive for the key-material implementation, but tracking docs had to be synchronized before continuing. Run 105 assumes the committed docs now correctly state the Run 104 boundary:

- full authority public key material exists and is hash-bound;
- MainNet requires complete authority material;
- the ratification verifier can verify using genesis material;
- ratification is not yet enforced on trust-bundle acceptance/apply paths;
- full C4 and C5 remain open.

Run 105 is the first enforcement run.

The goal is to require bundle-signing-key ratification on **non-mutating validation surfaces first**, without yet changing live apply mutation semantics.

---

# Exact objective

Implement the smallest safe enforcement layer that requires trust-bundle signing keys to be ratified by genesis-bound bundle-signing authority roots on non-mutating validation surfaces.

The objective is:

1. Define a ratification input path for trust-bundle validation.
2. Enforce ratification for bundle-signing keys on non-mutating validation surfaces.
3. Preserve existing signed-bundle verification, chain_id, environment, sequence, activation-height, activation_epoch, revocation, and minimum-margin checks.
4. Fail closed on unratified, malformed, wrong-chain, wrong-environment, unknown-root, transport-root, missing-key-material, or bad-signature ratification.
5. Preserve all non-mutation guarantees: no sequence write, no live trust mutation, no session eviction, no root merge on validation rejection.
6. Keep DevNet/TestNet behavior explicit and MainNet strict.
7. Do not implement rotation, revocation lifecycle, anti-rollback persistence, KMS/HSM, governance, or peer-driven live apply.

Run 105 should make this statement true for non-mutating validation paths:

> A trust bundle signed by an unratified bundle-signing key is rejected before any mutation side effect.

# Enforcement scope

Run 105 should prioritize non-mutating surfaces.

Required target surfaces:

## A. Startup preflight validation before mutation

Startup `--p2p-trust-bundle` path currently validates and then may sequence-write/root-merge/start networking.

Run 105 must ensure ratification validation happens before:

- sequence write;
- root merge;
- P2P trust start;
- network startup.

If ratification fails, startup must fail closed with no mutation side effect.

## B. Reload-check CLI

`--p2p-trust-bundle-reload-check` is validation-only.

Run 105 must enforce ratification here and preserve:

- no sequence write;
- no live trust mutation;
- no session eviction;
- no root merge.

## C. Peer-candidate check CLI

`--p2p-trust-bundle-peer-candidate-check` is validation-only.

Run 105 must enforce ratification here and preserve non-mutation.

## D. Live peer-candidate wire validation / propagation

If the live peer-candidate dispatcher can safely access the genesis-bound authority context and ratification object, enforce ratification before validation acceptance/rebroadcast.

If this surface cannot be safely wired without broad redesign, leave it explicitly open and return partial-positive. Do not fake it.

## E. Reload-apply / SIGHUP live reload

Run 105 may include reload-apply/SIGHUP validation only if the ratification check can be inserted before the existing `validate → snapshot → swap → evict_sessions → commit_sequence` mutation boundary without broad redesign.

If not, defer to Run 106 and document explicitly.

Do not weaken apply ordering.

# Ratification input model

Run 105 must decide and implement a narrow input model for the ratification object.

Acceptable options:

## Option 1 — Sidecar ratification file

A local ratification file is supplied alongside the trust bundle.

Possible shape:

```text
--p2p-trust-bundle <bundle.json>
--p2p-trust-bundle-ratification <ratification.json>
```

For check modes, analogous hidden/check-only flags may be added if consistent with repo style.

## Option 2 — Embedded ratification reference

If the trust-bundle schema already has a safe extension point, the bundle may refer to or include ratification data.

Do not change the trust-bundle wire format broadly.

## Option 3 — Genesis pre-ratified static initial bundle-signing key

If Run 100/101/104 already model a genesis initial bundle-signing key as pre-ratified, validation may use that explicit genesis authority state.

This must still be cryptographically/hash-bound and environment-bound.

Do not treat arbitrary local configured signing keys as ratified.

Choose the narrowest safe model and document the decision.

MainNet must not accept unratified local signing keys.
