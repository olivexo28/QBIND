# RUN 102 — Release-Binary Genesis Verification Wiring and Minimal Bundle-Signing-Key Ratification Verifier

## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

- Runs 050–068 completed signed PQC trust-bundle lifecycle.
- Runs 069–075 completed local operator reload lifecycle.
- Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
- Run 087 added peer-driven propagation/apply safety specification.
- Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
- Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
- Run 090 updated the operator runbook for propagation-only behavior.
- Runs 091–099 established, wired, and partially release-binary-evidenced canonical `activation_epoch` runtime-source behavior while preserving fail-closed behavior when no committed epoch exists.
- Run 100 added the formal production trust-anchor authority and bundle-signing-key ratification model:
  - production authority must be genesis-bound;
  - production static source-code anchors are rejected;
  - MainNet local/dev shortcuts are rejected;
  - bundle-signing-key ratification remains future implementation;
  - peer-driven live apply remains forbidden until authority ratification, anti-rollback, custody, emergency controls, and release-binary evidence exist.
- Run 101 added the additive genesis authority data model and canonical verification API:
  - `GenesisConfig.authority: Option<GenesisAuthorityConfig>`;
  - separate authority root structures for PQC transport roots and bundle-signing authority roots;
  - canonical genesis hash helper with domain separation;
  - `verify_boot_time_genesis` API with environment-specific fail-closed rules;
  - MainNet authority validation at the API/test level;
  - legacy genesis parsing preserved;
  - no source-code MainNet root anchors;
  - no transport-root reuse as bundle-signing authority.

Run 101 was partial-positive because the canonical verification API was not yet wired into the release binary’s real async startup path. The existing T233 `MainnetConfigError::ExpectedGenesisHashMissing` CLI shield remains in place, but full Run 101 boot-time verification is not yet enforced by the real binary startup flow.

Run 101 also left one operator-facing inconsistency: the release binary help text / operator surface around `--print-genesis-hash` still describes hashing the exact genesis file bytes, while Run 101 introduced a canonical parsed genesis hash over chain/environment/authority fields. Run 102 must correct this before relying on release-binary operator workflows.

Run 102 is the next implementation run.

It must first wire canonical genesis verification into the actual release-binary startup path and fix the canonical genesis hash operator surface. Only after that is safely done may it add the smallest minimal bundle-signing-key ratification verifier skeleton.

---

# Exact objective

Implement the next narrow authority-layer step after Run 101:

1. Wire Run 101’s canonical `verify_boot_time_genesis` into the real `qbind-node` release-binary startup path.
2. Make `--print-genesis-hash` print the canonical Run 101 genesis hash, not raw file-byte hash semantics.
3. Preserve existing DevNet/TestNet usability while enforcing strict MainNet fail-closed behavior.
4. Add the smallest minimal bundle-signing-key ratification verifier skeleton only if the release-binary genesis verification wiring is complete and stable.
5. Do not implement signing-key rotation, revocation lifecycle, authority anti-rollback persistence, KMS/HSM custody, governance, or peer-driven live apply.

Run 102 should close the Run 101 partial-positive gap: canonical genesis authority must be enforced by the actual release binary startup path, not just by library tests.

# Strict scope

Allowed:

## A. Wire boot-time genesis verification into real startup

Find the real release-binary startup path and call Run 101’s canonical verification before any authority-dependent behavior occurs.

Expected ordering:

```text
load genesis
→ canonicalize/hash parsed genesis
→ verify expected genesis hash according to environment policy
→ validate genesis authority fields according to environment policy
→ only then continue to trust-bundle processing / networking / consensus startup
```

MainNet must fail closed on:

- missing expected genesis hash;
- malformed expected genesis hash;
- genesis hash mismatch;
- missing required authority fields;
- malformed authority roots;
- wrong chain/environment binding if represented;
- authority validation failure.

Do not silently continue.

## B. Fix canonical genesis hash operator surface

Fix `--print-genesis-hash` or equivalent operator path so it prints the canonical Run 101 hash.

Requirements:

- hash must be computed from parsed/canonical genesis, not raw file bytes;
- hash must include chain_id, environment, validator set, and authority fields;
- help text must not claim raw file-byte hash semantics if that is no longer true;
- release-binary smoke evidence must prove authority field changes alter the printed hash.

## C. Preserve DevNet/TestNet behavior explicitly

DevNet may permit missing expected hash only as explicit non-production behavior.

TestNet policy must match the Run 101 / Run 100 model. If TestNet is not yet strict, document why and keep the boundary explicit.

MainNet must be strict.

## D. Minimal bundle-signing-key ratification verifier skeleton

Allowed only if A and B are complete.

The skeleton should define a narrow verifier surface that can answer:

```text
Is this bundle-signing key authorized by a genesis-bound authority root?
```

The skeleton may include:

- ratification object type;
- canonical signing preimage helper;
- domain separation tag;
- chain_id/environment checks;
- authority root fingerprint matching;
- signature-suite field validation;
- verifier API returning accepted/rejected with typed reasons.

But it must not implement rotation, revocation lifecycle, anti-rollback persistence, KMS/HSM custody, governance, peer-driven apply, or trust-bundle wire-format changes.

## E. Documentation and evidence

Update docs narrowly and honestly.
