  live inbound 0x05 validation-only

  staged peer-driven apply candidate

Mandatory safety model:
The spec must define a staged pipeline with at least these phases:

Phase 0 — receive
- peer sends candidate over live 0x05 frame;
- receiver authenticates transport peer through existing PQC transport;
- frame is size-bounded and decoded through existing peer-candidate envelope rules.

Phase 1 — validation-only
- existing Run 076/078/079/088/142/143 validation path runs;
- v2 candidates must pass Run 130 verifier and Run 132/142 marker validation-only check;
- invalid candidates are rejected and never rebroadcast;
- no mutation occurs.

Phase 2 — eligibility-to-stage
A peer-supplied candidate may be staged only if all of the following are true:
- validation-only result is accepted;
- candidate is newer or idempotent according to v2 marker discipline;
- trust-bundle signature verifies;
- chain_id, environment, genesis_hash, authority_root, and activation gates pass;
- Run 055 anti-rollback would accept if applied;
- local policy permits peer-driven staging;
- candidate was not seen recently, rate-limited, or suppressed;
- candidate does not conflict with a locally operator-pinned authority state;
- candidate is not below a locally persisted authority-domain sequence;
- candidate is not same-sequence/different-digest equivocation.

Phase 3 — local authorization gate
The spec must require an explicit local authorization decision before mutation. Define separate policies:
- DevNet: may allow auto-apply behind an explicit hidden devnet-only flag in a future run.
- TestNet: may allow auto-apply only with explicit operator opt-in and ratified v2 authority.
- MainNet: must require governance/ratification policy before any peer-driven apply; local peer majority alone is insufficient.

Phase 4 — apply
If eventually implemented, peer-driven apply must reuse the existing Run 070 ordering exactly:
  validate → snapshot previous → swap LivePqcTrustState → evict sessions → commit_sequence
and then apply v2 marker persistence only after sequence commit succeeds, following the same post-commit marker discipline used by Run 134/136/138.

Phase 5 — evidence and audit
The spec must require release-binary evidence before any closure claim.

Required invariants:
The future implementation must prove:
- no peer candidate can bypass local signature/ratification validation;
- no candidate can bypass v2 authority marker anti-rollback;
- no candidate can bypass Run 055 sequence anti-rollback;
- no candidate can bypass activation_height / activation_epoch gates;
- no peer majority can substitute for cryptographic ratification;
- no invalid candidate is propagated or applied;
- no same-sequence conflicting candidate is silently accepted;
- no lower-sequence candidate can downgrade local state;
- no stale snapshot or stale local data-dir can roll authority state backward;
- no peer-driven apply is enabled by default;
- MainNet refuses peer-driven apply without governance/ratification authority;
- apply uses existing Run 070 rollback/fatal semantics;
- marker persistence occurs only after sequence commit;
- session eviction occurs only after live trust swap succeeds and before sequence commit, preserving Run 070;
- rejected candidates do not write marker, sequence, metrics indicating apply, or live trust state;
- duplicate candidates do not form propagation loops;
- rate limits prevent peer-spam from causing unbounded validation cost;
- operator logs are stable enough for evidence harnesses.

Required threat model:
Document at least:
- malicious peer sends validly signed but unauthorized candidate;
- malicious peer sends lower-sequence rollback;
- malicious peer sends same-sequence different digest equivocation;
- malicious peer sends wrong-chain/wrong-environment/wrong-genesis candidate;
- malicious peer sends high-sequence candidate to pin future authority state;
- compromised but ratified bundle-signing key;
- compromised peer transport identity;
- network partition with two competing authority candidates;
- propagation loop / gossip amplification;
- DoS via large candidate or repeated candidates;
- stale snapshot restore followed by peer-driven apply;
- MainNet local-config-only authority bypass attempt;
- emergency revocation replay;
- operator accidentally enabling DevNet flag on MainNet.

Required policy matrix:
Create a table for DevNet/TestNet/MainNet covering:
- validation-only receive;
- propagation-only rebroadcast;
- peer-driven staging;
- peer-driven apply;
- required local flags;
- required ratification/governance proof;
- allowed default state;
- required evidence before promotion.

Expected stance:
- validation-only: allowed where already implemented.
- propagation-only: allowed only under existing disabled-by-default policy.
- peer-driven staging: future work, disabled by default.
- peer-driven apply: future work, disabled by default; MainNet blocked until governance/ratification/KMS-HSM assumptions are specified and evidenced.
