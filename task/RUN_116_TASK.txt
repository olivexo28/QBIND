# Exact objective

Design and, if safe within scope, implement the smallest durable authority anti-rollback persistence model for ratified bundle-signing authority state.

The objective is to prevent a node from accepting rollback to older or weaker bundle-signing authority state across:

1. process restart;
2. operator file rollback;
3. old ratification sidecar reuse;
4. snapshot restore;
5. data-dir copy/restore;
6. partial crash during authority update;
7. DevNet/TestNet/MainNet environment confusion.

Run 116 may be either:

- **spec-first only**, if the persistence design still needs careful modeling; or
- **implementation-narrow**, if the existing storage surfaces make a minimal safe persistent marker straightforward.

Prefer correctness over code volume. If there is uncertainty, deliver a strong design/evidence run and defer implementation.

Run 116 must not implement signing-key rotation/revocation lifecycle yet. It may define the persistence structure needed for those future runs.

---

# Core security problem

Current ratification checks prove that a bundle-signing key is authorized by genesis-bound authority material.

But without persistent anti-rollback state, a node may still be vulnerable to:

```text
accept authority sequence 10
restart / restore / file rollback
accept old authority sequence 7
accept bundle signed under older weaker authority path
```

or:

```text
accept ratification for key A at authority_epoch 5
later see stronger ratification state at authority_epoch 8
restore old data dir / old snapshot
re-accept authority_epoch 5 without detecting rollback
```

Run 116 must model and/or implement how the node records the highest accepted authority state so rollback is fail-closed.

# Strict scope

Allowed:

## A. Authority anti-rollback model

Define a persistent state model for ratified bundle-signing authority.

Expected conceptual fields:

```text
authority_sequence
or authority_epoch
genesis_hash
chain_id
environment
authority_root_fingerprint
ratified_bundle_signing_key_fingerprint
ratification_object_hash
ratification_valid_from / valid_until if used
status
last_update_source
```

Use repo-appropriate names and types.

## B. Persistence location analysis

Investigate and choose the correct persistence surface.

Candidates may include:

- existing `ConsensusStorage` / RocksDB metadata;
- existing trust-bundle sequence store;
- a new narrowly scoped authority-state file under `data_dir`;
- a new key under existing production consensus storage;
- a future on-chain authority state placeholder.

Do not create a new ad hoc persistence store if an existing durable, schema-versioned surface is appropriate.

## C. Monotonic rule definition

Define how the node compares authority states.

At minimum:

- accepted authority state must not decrease;
- same sequence with different ratification hash must fail closed unless explicitly allowed and justified;
- wrong chain/environment/genesis hash must fail closed;
- missing persisted state semantics must be environment-specific and safe;
- corruption must fail closed.

## D. Crash consistency model

Define or implement atomic update behavior.

The state must not allow:

- trust mutation without authority-state persistence when authority state should advance;
- authority-state persistence without corresponding validated authority transition, unless explicitly safe;
- partial writes that are accepted as valid on restart.

## E. Snapshot/restore interaction

Define how authority anti-rollback state interacts with snapshots.

At minimum:

- old snapshots must not silently roll back authority state;
- restoring a snapshot with older authority state than local persisted marker must fail closed or require explicit operator procedure;
- snapshot metadata requirements must be specified;
- Run 097 snapshot epoch parity lessons should be reused conceptually.

## F. Evidence docs and future run plan

If implementation is deferred, produce a concrete implementation plan for Run 117.

If implementation lands, produce tests and evidence.
