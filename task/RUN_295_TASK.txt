Backend design requirements:

1. Default Disabled / fail-closed

The production custody backend policy must default to Disabled.

Disabled policy must fail closed before any provider, transport, signing, or key-handle action.

No default runtime path may open or call the backend.

No fallback to fixture KMS, fixture HSM, RemoteSigner, local signing, dummy signing, or in-memory signing is allowed under production policy.

2. Provider class separation

Provider classes must be typed and explicit.

At minimum model:

* Disabled;
* FixtureKms;
* FixtureHsm;
* ProductionCloudKms;
* ProductionPkcs11Hsm;
* ProductionGenericKms;
* ProductionGenericHsm;
* Unknown / unsupported.

Fixture providers are allowed only for DevNet/TestNet source tests under explicit fixture policy.

Fixture providers must be refused for MainNet.

Production providers without configured provider identity/key handle/attestation must fail closed as unavailable or misconfigured, not silently fallback.

3. Domain binding

Every custody request and response must bind, where available:

* environment;
* chain_id;
* genesis hash or authority-domain binding;
* authority epoch or authority-domain sequence;
* custody provider kind;
* custody provider identity;
* key handle / key fingerprint;
* signer identity;
* request kind;
* authorized action;
* candidate/proposal/request digest;
* policy selector;
* custody protocol version;
* optional durable replay record digest;
* optional RemoteSigner transcript digest if composed;
* domain separation tag.

Wrong environment, wrong chain, wrong genesis/domain, wrong authority sequence, wrong provider kind, wrong provider identity, wrong key handle, wrong signer identity, wrong authorized action, wrong candidate digest, wrong policy, wrong protocol version, or wrong transcript must fail closed.

4. Canonical encoding

Use deterministic canonical bytes for all request/response/transcript digests.

Do not use Rust `Debug` formatting as canonical bytes.

Do not use wall-clock time as an authorizing value.

Do not use random request IDs unless they are explicitly non-authorizing and still bound into request/response correlation. Prefer deterministic request IDs in tests.

5. Request/response correlation

A custody response must only authorize the exact request it answers.

Reject:

* response for a different request id;
* response for a different transcript digest;
* response for a different candidate/proposal digest;
* response for a different authorized action;
* response for a different environment/chain/genesis;
* response from a wrong provider identity;
* response for a wrong key handle;
* response under wrong custody class;
* response signed/proved under an unexpected key;
* response missing signature/proof material;
* replayed response from a prior request.

6. Provider availability / error taxonomy

Surface typed fail-closed outcomes for:

* backend disabled;
* provider unavailable;
* provider misconfigured;
* provider endpoint unavailable;
* provider timeout;
* provider decode error;
* provider response too large;
* malformed response;
* unsupported provider kind;
* unsupported protocol version;
* wrong-domain response;
* wrong-key response;
* wrong-policy response;
* signing refused;
* key unavailable;
* attestation missing;
* attestation unavailable;
* attestation invalid / unverified;
* provider replay / stale nonce;
* ambiguous result;
* MainNet refused;
* MainNet production custody unavailable.

7. MainNet policy

MainNet must remain refused unless production authority criteria are satisfied.

Fixture KMS/HSM material must not satisfy MainNet.

RemoteSigner evidence alone must not satisfy MainNet KMS/HSM custody.

Local operator material alone must not satisfy MainNet.

Peer majority must not satisfy MainNet.

The backend must not allow a custody provider response alone to bypass custody attestation, governance proof verification, validator-set rotation requirements, or MainNet readiness gates.

8. No raw local production signing keys

The backend must not introduce raw local production signing-key loading.

Fixture keys may exist only in tests/examples and must be clearly non-production.

No secrets or provider credentials may be committed.
