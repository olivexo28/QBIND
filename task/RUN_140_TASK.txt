This is source/test wiring only. Release-binary snapshot/restore evidence is deferred to Run 141.

Background:
The existing v1 authority-marker restore path was already established in prior runs. Runs 131/134/136/138 added v2 marker primitives and v2 mutating-surface wiring for reload-apply, startup, and SIGHUP. Run 140 must extend restore parity to v2 using the existing versioned authority-marker record loader, not by creating a parallel marker format.

Implementation requirements:
1. Locate the existing snapshot/restore authority-marker v1 wiring:
   - restore marker copy/validation path;
   - any Run 122 / Run 124-style marker restore helper;
   - snapshot metadata and restore audit marker interactions;
   - current behavior for missing marker, v1 marker, corrupted marker, and mismatched domain.

2. Add v2 support using the existing versioned marker record type:
   - PersistentAuthorityStateRecordVersioned::V2 or the exact existing enum/type name;
   - existing v2 marker validation / load helpers;
   - existing domain tuple fields: environment, chain_id, genesis_hash, authority-domain sequence, marker digest/action fields.

3. Restore behavior must be fail-closed:
   - corrupted marker refuses restore or preserves existing fail-closed behavior;
   - wrong environment refuses;
   - wrong chain_id refuses;
   - wrong genesis hash refuses;
   - lower restored v2 authority-domain sequence than existing local marker refuses unless existing v1 semantics explicitly allow a documented restore case;
   - same-sequence different digest refuses;
   - v1-after-v2 downgrade refuses;
   - v2-after-v1 restore migration is allowed only if existing v2 marker rules allow it and sequence/domain checks pass.

4. Restore ordering:
   - Do not apply restored authority marker before snapshot validation succeeds.
   - Do not silently overwrite a newer local marker with an older restored marker.
   - Do not allow marker restore to bypass Run 055 trust-bundle sequence anti-rollback.
   - Do not treat marker restore as proof of trust-bundle validity by itself.
   - Snapshot marker restore must remain a metadata/state parity operation, not a new trust authorization path.

5. Crash-window discipline:
   - If VM/state snapshot restore succeeds but marker restore fails, the failure must be explicit and non-silent.
   - If existing v1 restore behavior already has a precise atomicity rule, mirror it for v2.
   - Do not delete or truncate existing marker files on failure.
   - Do not leave partially-written marker files or `.tmp` residue.

6. Audit/source tag:
   - Use the existing snapshot/restore authority update source variant if present.
   - Do not add a new AuthorityStateUpdateSource variant unless absolutely necessary.
   - If a new variant is necessary, explain why existing variants are insufficient and add tests proving serialization compatibility expectations.

Required test matrix:

A1. restore snapshot with v2 marker into empty data-dir
- Snapshot carries valid v2 marker.
- Restore succeeds.
- Restored marker is V2.
- latest_authority_domain_sequence matches snapshot marker.
- domain fields match runtime.
- audit/source field is restore-specific if the implementation records one.

A2. restore snapshot with v2 marker over same v2 marker
- Existing local marker and snapshot marker are identical.
- Restore succeeds/idempotent.
- Marker bytes unchanged or deterministically equivalent according to existing restore semantics.

A3. restore snapshot with higher-sequence v2 marker over older local v2 marker
- Existing marker seq=N.
- Snapshot marker seq=N+1.
- Restore succeeds only if this is consistent with existing restore anti-rollback policy.
- Marker advances to N+1.
- Document why this is safe under snapshot restore.

A4. restore v2 marker over v1 marker
- Existing v1 marker.
- Snapshot carries valid higher-sequence v2 marker.
- Restore succeeds as v2-after-v1 migration if existing marker rules allow it.
- Marker becomes V2.
- No downgrade path introduced.

R1. lower-sequence v2 restore rejected
- Existing local v2 marker seq=N.
- Snapshot marker seq<N.
- Restore fails closed.
- Existing marker bytes unchanged.
- No partial marker write.

R2. same-sequence different-digest v2 restore rejected
- Existing local v2 marker seq=N digest A.
- Snapshot marker seq=N digest B.
- Restore fails closed.
- Existing marker bytes unchanged.
- No partial marker write.

R3. wrong-environment v2 snapshot marker rejected
- Restore fails closed.
- Existing marker unchanged.
- No partial marker write.

R4. wrong-chain v2 snapshot marker rejected
- Restore fails closed.
- Existing marker unchanged.
- No partial marker write.

R5. wrong-genesis v2 snapshot marker rejected
- Restore fails closed.
- Existing marker unchanged.
- No partial marker write.

R6. corrupted v2 snapshot marker rejected
- Restore fails closed.
- Existing marker unchanged.
- Corrupted marker is not silently ignored or rewritten.

R7. v1 restore regression
- Existing v1 snapshot/restore marker behavior remains unchanged.
- No v2 path selected for v1 marker.
- v1 marker restored exactly as before.

R8. no-marker snapshot regression
- Snapshot without authority marker preserves existing behavior.
- No v2 marker fabricated.
- Restore behavior remains unchanged.

R9. v1-after-v2 downgrade refused
- Existing local marker is v2.
- Snapshot carries v1 marker.
- Restore refuses downgrade unless an existing, explicitly documented recovery rule already permits it.
- If refused, existing v2 marker bytes unchanged.
