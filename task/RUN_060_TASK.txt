4. Define normal rotation flows
   Produce step-by-step operator playbooks for:

   A. Normal transport root rotation
   - create new transport root offline
   - issue new leaf certs under new root
   - produce bundle sequence N+1 containing old root + new root during overlap
   - activate new root at height H if activation gating is used
   - roll validators to new leaf certs
   - produce bundle sequence N+2 marking old root retired
   - later produce bundle sequence N+3 revoking old root if needed
   - prove no rollback to N after N+1/N+2 due sequence persistence

   B. Emergency transport root revocation
   - produce sequence N+1 bundle revoking compromised root
   - ensure compromised-root leaf certs fail closed
   - operators replace affected certs
   - no static-root fallback
   - no Dummy crypto fallback
   - document liveness tradeoff if too many validators depend on revoked root

   C. Leaf certificate rotation
   - issue new leaf cert for validator
   - bundle does not necessarily need root change
   - deploy new leaf cert/KEM secret
   - revoke old leaf fingerprint if key compromise suspected
   - prove old revoked leaf fails closed

   D. Bundle-signing key rotation
   - because bundle-signing keys are configured via CLI today, define current safe operator path
   - if current code cannot rotate bundle-signing key via bundle itself, state that boundary honestly
   - recommended current process:
     - distribute new --p2p-trust-bundle-signing-key out of band
     - overlap old and new signing verification keys
     - publish bundle signed by new key
     - remove old signing key after rollout
   - do not pretend bundle-signing-key rotation is on-chain or self-updating unless implemented
