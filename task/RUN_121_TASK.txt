# Exact objective

Wire the Run 117/118/119 authority anti-rollback marker into the SIGHUP live reload path.

The objective is:

1. Derive `PersistentAuthorityStateRecord` only after SIGHUP ratification verification succeeds.
2. Compare the derived marker against the persisted marker before SIGHUP live reload mutation.
3. Reject conflict, corruption, wrong-domain, or invalid marker state before snapshot/swap/session eviction/sequence commit.
4. Persist the accepted marker atomically after the SIGHUP `commit_sequence` boundary.
5. Preserve Run 074/114/115 SIGHUP live reload ordering and repeated-trigger safety.
6. Keep startup behavior from Run 120 unchanged.
7. Keep process-start reload-apply behavior from Run 119 unchanged.
8. Do not persist from validation-only surfaces.
9. Do not invent a fake authority sequence.
10. Do not implement signing-key rotation/revocation lifecycle.
11. Do not implement peer-driven live apply.

Run 121 should make this statement true:

> SIGHUP live reload cannot silently accept a conflicting or corrupt authority marker; conflicts fail closed before live trust mutation, and accepted SIGHUP reloads persist the authority marker atomically after the existing SIGHUP sequence commit boundary.

# Strict scope

Allowed:

## A. SIGHUP marker wiring only

Wire only the long-running SIGHUP live reload path from Runs 074/114/115.

The path must:

- receive a SIGHUP trigger;
- re-read the configured candidate bundle / ratification sidecar as Run 114 required;
- validate and ratify the candidate;
- compare the authority marker before mutation;
- perform the existing live reload mutation only on acceptance;
- persist the authority marker after the existing sequence commit boundary.

Do not wire:

- validation-only reload-check;
- local peer-candidate check;
- live inbound `0x05` validation-only path;
- any peer-driven apply path;
- process-start reload-apply beyond regression preservation;
- startup beyond regression preservation.

## B. Marker derivation from verified ratification only

Use existing Run 118 / Run 119 helpers:

```text
derive_authority_state_from_ratification(...)
decide_marker_acceptance(...)
prepare_marker_for_acceptance(...)
persist_accepted_marker_after_commit_boundary(...)
```

or the repo-equivalent wrappers.

The marker must be derived only from verified ratification material.

Forbidden:

- deriving marker from unverified sidecar JSON;
- deriving marker from local config alone;
- deriving marker from trust-bundle sequence;
- deriving marker from activation height;
- deriving marker from activation_epoch.

## C. Compare before SIGHUP mutation

Before any SIGHUP live reload mutation:

- load persisted marker;
- fail closed on load/corruption/malformed marker;
- validate persisted marker domain against runtime environment, chain_id, and genesis_hash;
- compare candidate marker;
- reject conflict before mutation;
- do not repair, delete, or overwrite corrupt markers.

Rejected marker state must occur before:

- current-trust snapshot;
- live trust swap;
- session eviction;
- sequence commit;
- marker persistence.

## D. Persist marker after SIGHUP commit boundary

Use Run 117 atomic persistence through Run 119’s accept-and-persist helper.

The expected ordering is:

```text
trigger → validate/ratify → compare marker → snapshot current trust → swap live trust → evict_sessions → commit_sequence → persist_authority_marker
```

If a different ordering is used, the final report must justify why it is safer.

Do not claim cross-store atomicity unless it exists.

## E. Preserve repeated-trigger safety

SIGHUP can happen repeatedly on one long-running process.

Run 121 must preserve:

- invalid marker state followed by valid SIGHUP succeeds;
- valid SIGHUP followed by invalid marker state rejects without rollback;
- repeated invalid SIGHUPs do not mutate or advance sequence;
- repeated same valid candidate follows existing Run 055 anti-rollback behavior;
- marker persistence failure does not leave the controller in a poisoned state.

## F. Preserve DevNet policy

If Run 106 policy skips ratification in DevNet no-opt-in mode, SIGHUP behavior must remain unchanged and must not write an authority marker from unratified state.

MainNet/TestNet must enforce ratification and marker checks.

DevNet opt-in must enforce ratification and marker checks.
