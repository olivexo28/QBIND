Required design:
Add an additive, versioned governance-proof carrier to the live inbound `0x05` peer-candidate envelope or its existing v2 authority sidecar payload.

Preferred behavior:

* Existing live `0x05` v1 / legacy / no-proof v2 envelopes remain parseable.
* Existing no-proof envelopes remain accepted under `GovernanceProofPolicy::NotRequired`.
* Existing no-proof envelopes fail closed under `GovernanceProofPolicy::RequiredForLifecycleSensitive` for lifecycle-sensitive transitions.
* Proof-carrying envelopes produce `GovernanceProofContext::Available`.
* Malformed proof-carrying envelopes fail closed before propagation, staging, or apply.
* Invalid proof-carrying envelopes fail closed before propagation, staging, or apply.
* Live inbound `0x05` remains validation-only / staging-only according to existing policy; no apply-on-receipt is introduced.

1. Extend the live peer-candidate envelope with an optional governance-proof carrier, or reuse an existing nested v2 ratification sidecar field if safely available.
2. Convert the live-carried proof into the same `GovernanceAuthorityProof` / `GovernanceProofContext` used by Runs 163–175.
3. Route the resulting proof context into the existing validation-only marker decision path:

   * `preflight_v2_validation_only_marker_check_with_governance_proof_load`
   * or an equivalent live-envelope variant that shares the same governance gate.
4. Preserve `GovernanceProofPolicy::NotRequired` default behavior.
5. Honor the hidden Required-policy selector for live inbound `0x05` validation if the live path already receives policy context.
6. If the live path does not yet receive policy context, add the narrowest internal config plumbing needed at source/test level only.
7. Ensure invalid proof-carrying candidates are not propagated.
8. Ensure valid proof-carrying candidates are not applied automatically.
