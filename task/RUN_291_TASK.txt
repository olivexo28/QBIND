Backend design requirements:

The RocksDB durable replay backend must provide:

1. Domain binding

Persisted replay data must be bound to:

* environment;
* chain_id;
* genesis_hash or explicit optional genesis-domain binding if available;
* backend schema version;
* replay namespace / domain separator;
* authority-domain sequence or replay epoch if already part of the existing replay model.

Wrong environment, wrong chain, wrong genesis/domain, wrong namespace, or unsupported schema must fail closed.

2. Versioned schema

Define a stable schema version.

On open:

* supported schema opens successfully;
* absent empty DB initializes exactly once;
* unsupported future schema fails closed;
* malformed schema marker fails closed;
* missing required metadata in a non-empty DB fails closed;
* partial initialization residue fails closed or is recovered only if recovery is explicitly safe and tested.

3. Atomic record writes

A replay record write must be all-or-nothing.

If a record is acknowledged as written, it must survive reopen.

If a simulated write failure occurs before commit, no partial record may be visible.

If a simulated write failure occurs after an internal partial stage, recovery must fail closed or roll back deterministically.

4. Idempotency and equivocation

Duplicate identical replay record:

* must be idempotent;
* must not create a second logical record;
* must return a typed idempotent outcome.

Same replay record id with different digest:

* must fail closed as equivocation;
* must not overwrite the original record;
* must preserve the original durable record across reopen.

5. Replay ordering

If the existing durable replay model has sequence/order/stage constraints, preserve them.

Out-of-order records must fail closed unless the existing model explicitly permits them.

A later stage must not appear durable unless all required prior durable records exist or are supplied by the existing replay model.

6. Recovery / reopen semantics

Tests must prove:

* empty DB opens;
* write then reopen returns the same record;
* duplicate then reopen remains one record;
* equivocation attempt then reopen preserves the original record;
* wrong-domain reopen fails closed;
* unsupported schema reopen fails closed;
* corrupted metadata fails closed;
* corrupted record payload/digest fails closed;
* lock contention / second open fails closed if the existing RocksDB API surfaces lock errors;
* read-only or unwritable directory fails closed where representable.

7. No hidden fallback

The backend must never silently fall back to in-memory replay if RocksDB open/write/read fails.

There must be a typed unavailable/fail-closed outcome.

No `.tmp` residue or partial-state cleanup should silently erase evidence unless explicitly designed and tested.

8. Production default policy

Default policy must remain Disabled or unavailable/fail-closed.

Fixture/in-memory replay remains available only where already allowed.

Production RocksDB backend is source/test selectable in this run but not default-enabled in the production binary.

MainNet must not be enabled by this run.
