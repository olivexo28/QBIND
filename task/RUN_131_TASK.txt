# Exact objective

Extend the authority marker model to represent ratification v2 monotonic state and implement the v1→v2 migration primitive, without wiring v2 into production enforcement surfaces.

The objective is:

1. Add a versioned marker model capable of representing both current v1 marker state and new v2 monotonic state.
2. Add v2 marker fields required by Run 129/130:
   - `authority_schema_version` or equivalent;
   - `latest_authority_domain_sequence`;
   - `latest_lifecycle_action`;
   - active/current key fingerprint;
   - previous key fingerprint where applicable;
   - latest v2 ratification digest;
   - optional revoked-key digest/metadata placeholder if specified.
3. Implement pure comparison rules for v2 markers:
   - first v2 marker;
   - same sequence / same digest idempotent;
   - same sequence / different digest rejected;
   - lower sequence rejected;
   - higher sequence accepted;
   - wrong domain rejected;
   - wrong key/action linkage rejected.
4. Implement v1→v2 migration primitive:
   - v2-after-v1 is allowed only through explicit migration rules;
   - v1-after-v2 is fail-closed;
   - v1 marker remains parse-compatible;
   - v2 marker remains deterministic and fail-closed on corruption.
5. Extend snapshot metadata representation if needed so v2 marker state can be carried later.
6. Preserve all v1 marker behavior and tests.
7. Do not wire v2 marker comparison into startup/reload/SIGHUP/reload-check/peer-candidate/live `0x05` yet.
8. Do not implement signing-key rotation/revocation lifecycle yet.

Run 131 should make this statement true:

> QBIND can represent, serialize, parse, digest, compare, and migrate authority marker state from v1 to v2 monotonic semantics, but no production surface consumes v2 marker logic yet.
