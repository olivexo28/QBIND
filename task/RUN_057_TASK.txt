3. Define activation semantics
   Recommended policy:
   - activation_height is inclusive:
     - current_height >= activation_height means active
     - current_height < activation_height means fail closed / not yet active
   - activation_epoch is inclusive:
     - current_epoch >= activation_epoch means active
     - current_epoch < activation_epoch means fail closed / not yet active
   - If both height and epoch gates are present:
     - both must be satisfied.
   - Missing gate:
     - no activation restriction from that field.
   - Invalid gate:
     - fail closed.
   - Bundle/root with future activation:
     - must not be merged into live root set.
   - Bundle/root with satisfied activation:
     - behaves as before.
   - Activation check must happen before sequence persistence update if accepting a future bundle would incorrectly burn a higher sequence.
     - Important: do not persist sequence for a bundle that is not yet active.
   - Activation check must happen before roots are merged.

4. Define sequence interaction
   Required:
   - A not-yet-active bundle must not update the highest accepted sequence.
   - A rejected future-activation bundle must leave the existing persisted sequence record unchanged.
   - If a lower sequence is not yet active, rollback still should be rejected according to existing ordering if sequence is checked before activation; choose and document ordering.
   - Recommended ordering:
     1. parse bundle
     2. verify signature
     3. validate env/chain_id/root status/revocation/windows
     4. activation gating
     5. sequence monotonicity persistence
     6. merge roots
   - This prevents not-yet-active future bundles from advancing sequence.
