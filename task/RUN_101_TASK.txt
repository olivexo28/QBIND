# RUN 101 — Genesis Authority Fields and Boot-Time Genesis Hash Binding

## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

- Runs 050–068 completed signed PQC trust-bundle lifecycle.
- Runs 069–075 completed local operator reload lifecycle.
- Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
- Run 087 added peer-driven propagation/apply safety specification.
- Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
- Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
- Run 090 updated the operator runbook for propagation-only behavior.
- Runs 091–099 established, wired, and release-binary-evidenced the canonical activation_epoch runtime-source path, while preserving fail-closed behavior when no committed epoch exists.
- Run 100 added the formal production trust-anchor authority and bundle-signing-key ratification model:
  - production authority must be genesis-bound
  - production static source-code anchors are rejected
  - MainNet local/dev shortcuts are rejected
  - bundle-signing-key ratification remains future implementation
  - peer-driven live apply remains forbidden until authority ratification, anti-rollback, custody, emergency controls, and release-binary evidence exist

Run 100 was spec-first and docs-only. It did not implement genesis authority fields, boot-time genesis hash binding, a ratification verifier, KMS/HSM custody, governance, or peer-driven live apply.

Run 101 is the first implementation step after Run 100.

The goal is to add the **canonical genesis-bound authority root surface** and boot-time hash binding required by the Run 100 model.

Run 101 must not implement full signing-key ratification yet.

---

# Exact objective

Implement the smallest additive genesis authority fields and boot-time genesis hash verification needed to bind initial production trust-anchor authority to canonical genesis configuration.

The objective is:

1. Extend the existing genesis configuration with additive authority fields.
2. Canonically serialize/hash the genesis configuration.
3. Verify boot-time expected genesis hash when configured/required.
4. Bind initial PQC trust-anchor authority and bundle-signing authority root to:
   - chain_id
   - environment
   - genesis_hash
5. Enforce strict MainNet refusal rules if required authority material or expected hash is missing/mismatched.
6. Preserve DevNet/TestNet usability with explicit non-production allowances.
7. Do not implement full bundle-signing-key ratification yet.
8. Do not change trust-bundle wire format.
9. Do not implement peer-driven live apply.

Run 101 should establish the genesis authority root that later Runs 102–103 can use for bundle-signing-key ratification and anti-rollback persistence.

---

# Architectural decision

Run 101 must follow the Run 100 decision:

## Production initial authority comes from genesis configuration bound by boot-time cryptographic hash.

Do NOT bake production root anchors directly into Rust source-code static constants.

Allowed:

- test-only fixtures
- DevNet helper-generated materials
- explicit non-production local shortcuts
- hidden evidence-only flags if already consistent with repo style

Forbidden:

- MainNet production static root constants
- hidden fallback anchors
- hidden fallback signing keys
- accepting authority not bound to genesis
- accepting wrong-chain or wrong-environment authority
- treating local config alone as sufficient MainNet authority

## C. Boot-time expected genesis hash verification

Add boot-time verification that compares computed canonical genesis hash against an expected hash when required.

Expected behavior:

* MainNet: expected genesis hash required
* TestNet: expected genesis hash strongly required or explicitly configured by environment policy

Failure behavior:

* missing required expected hash: fail closed
* mismatch: fail closed
* malformed hash: fail closed
* wrong chain/environment: fail closed

Do not silently continue.
