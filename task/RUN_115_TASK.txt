# Exact objective

Produce release-binary evidence proving that SIGHUP live reload enforces bundle-signing-key ratification before any live reload mutation.

Run 115 should prefer **no production runtime code changes**.

Allowed changes are limited to:

- DevNet evidence scripts;
- evidence-only fixture/helper generation;
- log/metrics assertion helpers;
- documentation/evidence updates;
- narrow bug fixes if release-binary evidence exposes a real Run 114 implementation defect.

The objective is to prove on real `target/release/qbind-node` binaries:

1. Valid ratification allows SIGHUP live reload to apply.
2. Missing ratification rejects before mutation.
3. Bad ratification rejects before mutation.
4. Wrong-chain / wrong-environment / wrong-root ratification rejects before mutation if covered by the harness.
5. Rejection occurs before:
   - current trust snapshot;
   - live trust-state swap;
   - session eviction;
   - sequence commit;
   - sequence-file write;
   - root merge.
6. Successful SIGHUP preserves Run 074 ordering:

```text
trigger → validate → snapshot current trust → swap live trust → evict_sessions → commit_sequence
```

7. Repeated-trigger safety is proven:
   - invalid → valid succeeds;
   - valid → invalid rejects without rollback;
   - repeated invalids do not mutate or advance sequence;
   - repeated same valid candidate follows existing sequence anti-rollback behavior.
8. DevNet policy remains explicit:
   - no-opt-in behavior preserved if intentionally allowed;
   - opt-in enforces ratification.
9. No peer-driven live apply is introduced.
10. No production runtime design is changed.

Produce logs, exit codes, metrics snapshots if applicable, and a summary for each scenario.

## Scenario 1 — Valid ratification allows SIGHUP reload

Use a running node and a SIGHUP reload candidate signed by a ratified bundle-signing key with valid ratification sidecar.

Prove:

- SIGHUP trigger is observed;
- ratification verifies;
- existing trust-bundle checks still run;
- live reload proceeds;
- ordering remains:

```text
trigger → validate → snapshot current trust → swap live trust → evict_sessions → commit_sequence
```

- sequence commit occurs only after swap/evict;
- no bypass of activation/revocation/sequence checks.

## Scenario 2 — Missing ratification rejects before mutation

Omit ratification sidecar or remove it before SIGHUP under MainNet/TestNet strict policy or DevNet opt-in strict policy.

- SIGHUP trigger is observed;
- ratification rejection appears;
- no current-trust snapshot;
- no live trust swap;
- no session eviction;
- no sequence commit;
- no sequence-file write;
- no root merge;
- node remains alive for subsequent SIGHUP.

## Scenario 3 — Bad ratification rejects before mutation

Use bad signature, wrong chain, wrong environment, or unknown authority root.

Prove same negative invariants as Scenario 2.

## Scenario 4 — Repeated-trigger safety

- invalid SIGHUP followed by valid SIGHUP succeeds;
- valid SIGHUP followed by invalid SIGHUP rejects without rolling back valid state;
- repeated invalid SIGHUPs do not mutate or advance sequence;
- repeated same valid candidate follows existing sequence anti-rollback behavior.

## Scenario 5 — DevNet policy behavior
