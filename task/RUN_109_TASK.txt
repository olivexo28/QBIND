# Exact objective

Implement ratification enforcement for live inbound peer-candidate `0x05` validation.

The objective is:

1. When a running node receives a peer-candidate `0x05` frame, validate the candidate using the same ratification policy and verifier stack established in Runs 103–108.
2. Enforce MainNet/TestNet default-strict ratification.
3. Preserve DevNet explicit opt-in behavior.
4. Reject unratified, missing, malformed, wrong-chain, wrong-environment, bad-signature, unknown-root, transport-root, unsupported-suite, and missing-key-material ratification cases before validation success.
5. Ensure rejected/unratified candidates are not rebroadcast by the Run 088 propagation path.
6. Preserve all existing live peer-candidate non-mutation guarantees:
   - no apply;
   - no sequence write;
   - no live trust mutation;
   - no root merge;
   - no session eviction;
   - no consensus ratification.
7. Do not change the peer-candidate wire format.
8. Do not implement peer-driven live apply.
9. Do not touch reload-apply or SIGHUP in this run.

Run 109 should make this statement true:

> Live inbound peer-candidate frames signed by unratified bundle-signing keys fail validation and are never rebroadcast.
