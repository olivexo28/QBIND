# RUN 096 — Canonical Binary-Path Reconfig Proposal Source for Release-Binary Epoch Transition Evidence

## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

* Runs 050–068 completed signed PQC trust-bundle lifecycle.
* Runs 069–075 completed local operator reload lifecycle.
* Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
* Run 087 added peer-driven propagation/apply safety specification.
* Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
* Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
* Run 090 updated the operator runbook for propagation-only behavior.
* Run 091 established fail-closed `CurrentEpochUnavailable` behavior for `activation_epoch` whenever no canonical runtime epoch source exists.
* Run 092 proved that production `qbind-node` did not open `RocksDbConsensusStorage`, so `meta:current_epoch` was phantom on real binary runs.
* Run 093 introduced canonical production binary-path `ConsensusStorage` ownership at `<data_dir>/consensus`.
* Run 094 threaded the Run 093 `ConsensusStorage` handle into the binary consensus loop and added fail-closed epoch persistence through existing atomic epoch persistence machinery.
* Run 095 added binary-path committed reconfig-block detection and engine epoch transition trigger:

  * detects existing canonical reconfig representation
  * uses `BlockHeader.payload_kind == PAYLOAD_KIND_RECONFIG`
  * uses `BlockHeader.next_epoch`
  * calls existing `BasicHotStuffEngine::transition_to_epoch(...)`
  * threads actual committed reconfig block ID into the Run 094 persistence helper
  * fails closed if a real transition lacks a committed reconfig block ID

Run 095 was partial-positive because the production binary still cannot honestly create or propose a canonical reconfig block. The binary leader currently emits only normal blocks, effectively:

```rust
payload_kind = PAYLOAD_KIND_NORMAL
next_epoch = 0
```

Therefore release-binary Scenario 2 and Scenario 3 remain unproven:

* real committed reconfig transition
* persisted `CommittedEpoch(n)`
* restart observing `CommittedEpoch(n)`

Run 096 must address only that missing piece:

> Provide the smallest canonical production-binary source of reconfig proposals so release binaries can commit a real `PAYLOAD_KIND_RECONFIG` block, trigger Run 095 epoch transition, trigger Run 094 persistence, and prove restart sees `CommittedEpoch(n)`.

---

# Exact objective

Implement the smallest safe binary-path mechanism for proposing a canonical reconfig block under explicit local operator control, suitable for DevNet/TestNet evidence and future governance integration.

The objective is:

1. Add a tightly gated binary-path source of reconfig proposal intent.
2. Ensure the leader can propose a block with:

   * `payload_kind = PAYLOAD_KIND_RECONFIG`
   * canonical `next_epoch`
   * no synthetic epoch derivation
   * no wall-clock/height/view-derived epoch
3. Ensure the reconfig block commits through the existing HotStuff path.
4. Ensure Run 095 detects the committed reconfig block.
5. Ensure Run 095 calls `transition_to_epoch(...)`.
6. Ensure Run 094 persists the transition through `apply_epoch_transition_atomic(...)`.
7. Ensure restart observes `CommittedEpoch(n)` through the Run 093 storage open surface.
8. Preserve Run 091/092 fail-closed trust-bundle activation behavior.

Run 096 must make release-binary end-to-end epoch transition evidence possible.

Run 096 must **not** consume `current_epoch` for trust-bundle activation.

# Strict scope

Implement a minimal local operator-controlled mechanism to supply reconfig proposal intent to the binary leader.

Acceptable forms, if consistent with existing repo patterns:

* hidden DevNet/TestNet-only CLI flag
* local file-based reconfig proposal input
* existing genesis/config-driven reconfig input if already present

It must not be peer-driven.

It must not be automatic.

It must not infer epoch from time, height, view, or progress.

---

## B. Canonical reconfig block construction

When the local leader is authorized to propose reconfig:

* construct the block using existing block/proposal structures
* set existing canonical reconfig fields only
* `payload_kind = PAYLOAD_KIND_RECONFIG`
* `next_epoch = target_epoch`
* preserve existing block ID derivation
* preserve existing consensus signing/preimage logic
* preserve existing HotStuff commit rules

Do not introduce a parallel block format.

---

## C. Validation before proposal

Before a node proposes a reconfig block, validate:

* target epoch is present
* target epoch is strictly greater than current engine epoch
* target epoch is not derived from wall-clock/height/view
* proposal source is local/operator-gated
* environment restrictions are obeyed
* reconfig proposal is not silently enabled on MainNet unless an existing MainNet governance path already authorizes it

If any check fails, fail closed.
