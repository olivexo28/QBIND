# Exact objective

Wire the Run 117/118/119 authority anti-rollback marker into the startup `--p2p-trust-bundle` acceptance path.

The objective is:

1. Derive `PersistentAuthorityStateRecord` only after startup trust-bundle ratification verification succeeds.
2. Compare the derived marker against the persisted marker before startup trust mutation, sequence/root persistence, or network startup.
3. Reject conflict, corruption, wrong-domain, or invalid marker state before any startup mutation.
4. Persist the accepted marker atomically after the accepted startup commit boundary.
5. Preserve all existing startup validation and ordering invariants.
6. Keep process-start reload-apply behavior from Run 119 unchanged.
7. Do not wire SIGHUP in this run.
8. Do not persist from validation-only surfaces.
9. Do not invent a fake authority sequence.
10. Do not implement signing-key rotation/revocation lifecycle.
11. Do not implement peer-driven live apply.

Run 120 should make this statement true:

> Startup `--p2p-trust-bundle` acceptance cannot silently accept a conflicting or corrupt authority marker; conflicts fail closed before startup mutation/network startup, and accepted startup mutations persist the authority marker atomically after the existing startup commit boundary.

# Strict scope

Allowed:

## A. Startup trust-bundle marker wiring only

Wire only the startup path that accepts an initial `--p2p-trust-bundle` and may:

- validate the signed bundle;
- enforce ratification;
- persist sequence/root state;
- merge accepted trust roots;
- start P2P/networking with accepted trust material.

Do not wire:

- SIGHUP live reload;
- validation-only reload-check;
- local peer-candidate check;
- live inbound `0x05` validation-only path;
- process-start reload-apply beyond regression preservation.

## B. Marker derivation from verified ratification only

Use Run 118 / Run 119 helpers:

```text
derive_authority_state_from_ratification(...)
decide_marker_acceptance(...)
prepare_marker_for_acceptance(...)
```

or the repo-equivalent wrappers.

The marker must be derived only from verified ratification material.

Forbidden:

- deriving marker from unverified sidecar JSON;
- deriving marker from local config alone;
- deriving marker from trust-bundle sequence;
- deriving marker from activation height;
- deriving marker from activation_epoch.

## C. Compare before startup mutation

Before any startup trust mutation or network start:

- load persisted marker;
- fail closed on load/corruption/malformed marker;
- validate persisted marker domain against runtime environment, chain_id, and genesis_hash;
- compare candidate marker;
- reject conflict before mutation;
- do not repair, delete, or overwrite corrupt markers.

## D. Persist marker after accepted startup commit boundary

Use Run 117 atomic persistence through Run 119’s accept-and-persist helper.

Run 119 selected the pattern:

```text
validate/ratify → compare marker → perform existing mutation/commit → persist marker after commit boundary
```

For startup, identify the equivalent startup commit boundary and persist after it.

If startup has multiple commits, choose the safest point and document the crash behavior clearly.

Do not claim cross-store atomicity unless it exists.

## E. Preserve startup behavior when ratification is not invoked

If Run 106 policy skips ratification in DevNet no-opt-in mode, startup behavior must remain unchanged and must not write authority marker from unratified state.

MainNet/TestNet must enforce ratification and marker checks.
