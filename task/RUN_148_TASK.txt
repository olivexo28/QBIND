Task: Run 148 — Source/Test DevNet-Only Peer-Driven Apply Using Existing Run 070 Contract

Current accepted state:
- Run 144: peer-driven trust-bundle apply safety specification landed.
- Run 145: non-applying PeerCandidateStagingQueue scaffold landed.
- Run 146: staging queue hook wired into live inbound 0x05 dispatcher after validation acceptance.
- Run 147: hidden disabled-by-default DevNet/TestNet-only staging arming flag added and release-binary evidence captured.
- Current peer-candidate staging is non-authoritative and non-applying.
- MainNet staging remains refused.
- Peer-driven live apply remains unimplemented.

Objective:
Implement the first source/test-only peer-driven live apply path, limited to DevNet/TestNet, disabled by default, local-policy gated, and strictly reusing the existing Run 070 apply contract.

This run must allow an already-staged, already-validated peer candidate to be applied only when an explicit local peer-driven apply policy is enabled. It must not introduce a new apply algorithm. It must reuse the existing validation → snapshot previous → swap LivePqcTrustState → evict sessions → commit_sequence ordering and the existing v2 post-commit marker persistence discipline.

Strict scope:
- Source/test wiring only.
- No release-binary evidence in this run.
- DevNet/TestNet only.
- MainNet refused unconditionally.
- Disabled by default.
- No governance implementation.
- No KMS/HSM implementation.
- No signing-key rotation/revocation lifecycle.
- No new wire format.
- No trust-bundle schema change.
- No ratification sidecar schema change.
- No marker schema change.
- No sequence-file schema change.
- No automatic apply on validation alone.
- No apply of unstaged candidates.
- No apply of invalid candidates.
- No apply of expired staged candidates.
- Do not weaken existing local reload-apply, SIGHUP, startup, snapshot/restore, validation-only, staging, or propagation behavior.
- Do not claim MainNet readiness.
- Do not claim full C4 or C5 closure.

Design requirement:
Peer-driven apply must be a separate local decision after staging.

Required pipeline:
1. live inbound 0x05 candidate received;
2. existing validation-only path accepts;
3. Run 146 staging hook stages candidate;
4. explicit local peer-driven apply policy is evaluated;
5. staged candidate is revalidated or validated candidate material is proven fresh;
6. existing Run 070 apply contract is invoked;
7. v2 authority marker is persisted only after commit_sequence succeeds;
8. failure semantics mirror Run 070 / Run 134 / Run 138.

Implementation requirements:

1. Add explicit apply policy type.

Prefer a new internal config type:

PeerDrivenApplyPolicy {
    enabled: bool,
    environment: Environment,
    allow_devnet: bool,
    allow_testnet: bool,
    allow_mainnet: bool,
    require_staged_candidate: bool,
    require_v2_ratification: bool,
    max_candidate_age_secs: u64,
}

Required defaults:
- enabled=false;
- allow_devnet=false unless explicitly set by tests/config;
- allow_testnet=false unless explicitly set by tests/config;
- allow_mainnet=false always;
- require_staged_candidate=true;
- require_v2_ratification=true.

2. Add an apply controller.

Prefer a small module:

crates/qbind-node/src/pqc_peer_candidate_apply.rs

It should expose a narrow API such as:

try_apply_staged_peer_candidate(
    staged_candidate_id,
    staging_queue,
    apply_context,
    marker_config,
    policy,
    runtime_domain,
    now
) -> PeerDrivenApplyOutcome

Outcome variants should distinguish:
- Disabled;
- RefusedMainNet;
- CandidateNotFound;
- CandidateExpired;
- CandidateNotValidated;
- CandidateWrongDomain;
- CandidateMarkerConflict;
- ApplySucceeded;
- ApplyRejected;
- ApplyRollbackSucceeded;
- ApplyFatalRollbackFailed;
- MarkerPersistedAfterCommit;
- MarkerPersistFailedAfterCommit.

3. Reuse Run 070 apply contract.

The implementation must call the existing apply path, not duplicate logic:
- `apply_validated_candidate_with_previous` or exact existing production apply helper;
- existing `LiveTrustApplyContext`;
- existing session eviction path;
- existing sequence commit path.

The future peer-driven apply controller must not directly mutate:
- `LivePqcTrustState`;
- sequence file;
- authority marker file;
except through the existing apply and marker persistence helpers.

4. Reuse v2 marker post-commit discipline.

For v2 candidates:
- marker decision/check occurs before apply;
- no marker write before apply;
- marker persist occurs only after sequence commit succeeds;
- marker persist failure after successful sequence commit must be fatal/operator-actionable, following Run 134/138 discipline.

5. MainNet refusal.

MainNet must refuse peer-driven apply even if policy enabled. The refusal must be typed and tested. Do not add a MainNet bypass flag.

6. Candidate freshness.

Expired staged candidates must not apply. TTL must be enforced at apply time, not only staging time.

7. Authorization boundary.

DevNet/TestNet local policy may authorize apply for this run. MainNet must require future governance/ratification/KMS-HSM policy and remain blocked.
