# Exact objective

Implement the smallest backward-compatible snapshot epoch parity support so that snapshots created after a committed epoch transition carry the canonical committed epoch, and restore can preserve that epoch into the production consensus-storage surface.

The objective is:

1. Add an additive optional epoch field to `StateSnapshotMeta`.
2. Populate that field during snapshot creation using the canonical committed epoch source.
3. Validate the epoch field during snapshot validation without breaking old snapshots.
4. Restore the epoch into canonical production `ConsensusStorage` when restoring from a snapshot that contains epoch metadata.
5. Preserve explicit ambiguity for old snapshots that lack epoch metadata.
6. Preserve Run 091/092 fail-closed `CurrentEpochUnavailable` behavior for trust-bundle activation.
7. Do not consume `current_epoch` for trust-bundle activation in Run 097.

Run 097 should make snapshot/rejoin epoch state honest and durable.

Run 097 must not yet use the restored epoch to satisfy trust-bundle activation.

# Strict scope

Allowed:

## A. Additive snapshot metadata field

Extend `StateSnapshotMeta` with an optional epoch field.

Expected shape:

```rust
epoch: Option<u64>
```

or the repo’s existing epoch type if one already exists and is appropriate.

Requirements:

* backward-compatible with existing snapshot metadata
* old snapshots without epoch remain valid
* new snapshots may include epoch
* absence of epoch must remain explicit, not coerced to `0`

---

## B. Snapshot creation epoch population

When creating a snapshot from a node with a canonical committed epoch available, write the epoch into snapshot metadata.

The epoch must come from a canonical source only, such as:

* Run 093/094/096 production `ConsensusStorage` committed epoch
* existing engine committed/current epoch if already canonical at snapshot time

Do not derive epoch from:

* block height
* wall-clock time
* view number
* timer ticks
* snapshot height
* filename
* directory name

---

## C. Snapshot validation

Update snapshot validation so that:

* old metadata without epoch remains accepted
* new metadata with epoch is parsed and validated
* malformed epoch fails closed
* epoch `0` is accepted only if it is semantically valid in existing repo epoch rules; otherwise reject or preserve explicit no-epoch state
* no missing epoch is silently treated as `0`

---

## D. Restore path epoch parity

When restoring from a snapshot that includes epoch metadata:

* materialize VM-v0 state exactly as before
* open/use canonical production `ConsensusStorage`
* persist or restore `meta:current_epoch` consistently with the snapshot epoch
* preserve atomicity/fail-closed behavior
* avoid partial restore where VM state is restored but epoch metadata fails silently

When restoring from an old snapshot without epoch metadata:

* restore must remain backward-compatible
* epoch state must remain explicit unknown/no-epoch
* no synthetic epoch must be written

---

## E. Trust activation isolation

Every production `ActivationContext` must continue to use:

```rust
current_epoch: None
```

Run 097 must not feed snapshot epoch or restored consensus-storage epoch into trust-bundle activation.

* backward-compatible

## 2. Populate epoch on snapshot creation

When a canonical committed epoch is available:

* include it in `meta.json`
* record source in logs/evidence where useful

* write `epoch: null` or omit field according to compatibility style
* do not fake it

---

## 3. Validate epoch metadata

Validation must:

* accept old snapshots without epoch
* accept valid new snapshots with epoch
* reject malformed/incompatible epoch metadata
* preserve existing chain-id/height/block-hash validation

---

## 4. Restore epoch into consensus storage

When snapshot metadata includes epoch:

* write/persist it through existing canonical storage machinery
* prefer existing atomic transition/MetaStore APIs
* preserve storage consistency checks
* fail closed on write/recovery failure

When metadata lacks epoch:

* do not write epoch
* preserve explicit no-epoch state

---

## 5. Preserve restore atomicity

Avoid a partial restore where:

* VM state is restored
* epoch restore fails
* node continues anyway

If epoch metadata exists but cannot be restored, startup must fail closed.

---

No `ActivationContext.current_epoch = Some(...)` in Run 097.
