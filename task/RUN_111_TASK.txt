# Exact objective

Execute and archive the Run 110 N=3 release-binary live peer-candidate ratification evidence.

The objective is to prove on real release binaries:

1. Valid ratified live `0x05` candidate validates on the relay.
2. Valid ratified candidate propagates under existing Run 088 rules when propagation is enabled.
3. Source-peer exclusion remains intact.
4. Missing/unratified candidate rejects at the ratification gate.
5. Missing/unratified candidate does not propagate.
6. Bad/tampered ratification rejects at the correct boundary.
7. Duplicate rejected candidate cannot become accepted or rebroadcast through seen-cache behavior.
8. No apply occurs.
9. No sequence write occurs.
10. No live trust mutation occurs.
11. No root merge occurs.
12. No session eviction occurs.
13. No consensus ratification occurs.
14. DevNet no-opt-in legacy behavior remains explicitly non-production if covered by the Run 110 harness.

Run 111 should prefer **no code changes**.

- fixing the existing Run 110 harness if it has a narrow execution bug;
- fixing evidence-only fixture/helper issues;
- adding log assertions if the existing harness is under-asserting;
- adding documentation/evidence files;
- updating docs to record results.

No production runtime behavior should change.

Produce logs and assertions for each scenario covered by the harness.

## Scenario 1 — Baseline ratification startup

## Scenario 2 — Valid ratified live candidate validates and propagates

Prove:

- V1 receives valid ratified candidate from V0;
- V1 accepts validation;
- V1 rebroadcasts to V2 when propagation enabled;

## Scenario 6 — DevNet no-opt-in legacy behavior

If present, prove DevNet no-opt-in preserves explicitly documented legacy behavior and does not imply MainNet/TestNet behavior.
