Backend design requirements:

1. Default Disabled / fail-closed

RemoteSigner backend policy must default to Disabled.

Disabled policy must fail closed before any network/transport/signing attempt.

No default runtime path may open or call the backend.

No silent fallback to fixture, loopback, local signing, or in-memory signing is allowed.

2. Domain binding

Every RemoteSigner request and response must bind, where available:

* environment;
* chain_id;
* genesis hash or authority-domain binding;
* authority-domain sequence or authority epoch;
* signer identity / signer key fingerprint;
* request kind;
* authorized action;
* candidate digest or proposal digest;
* replay/durable record digest if composed;
* policy selector;
* nonce/request id if used;
* backend protocol version;
* domain separation tag.

Wrong environment, wrong chain, wrong genesis/domain, wrong authority sequence/epoch, wrong signer identity, wrong request id, wrong authorized action, wrong candidate digest, wrong payload digest, wrong policy, or wrong protocol version must fail closed.

3. Canonical encoding

Use deterministic canonical bytes for all transcript digests.

Do not use Rust `Debug` formatting as canonical bytes.

Do not use wall-clock time as an authorizing value.

Do not use random request IDs unless they are explicitly non-authorizing and still bound into request/response correlation. Prefer deterministic request IDs derived from typed input for tests.

4. Request/response correlation

A RemoteSigner response must only authorize the exact request it answers.

The backend must reject:

* response for different request id;
* response for different transcript digest;
* response for different candidate digest;
* response for different authorized action;
* response for different chain/environment/genesis;
* response from wrong signer identity;
* replayed response from a prior request;
* response signed/proved under an unexpected key;
* response with missing or malformed proof/signature material.

5. Transport / availability

The production backend must surface typed fail-closed outcomes for:

* endpoint unavailable;
* connection refused;
* timeout;
* transport decode error;
* response too large;
* malformed response;
* unsupported protocol version;
* response-domain mismatch;
* response-transcript mismatch;
* signer unavailable;
* signer refused;
* signer policy rejected;
* signer key unavailable;
* signer attestation missing or unavailable if the existing surface models attestation.

Use existing transport abstractions from Runs 201–202 if available. If a new narrow transport is necessary, keep it minimal and mockable.

6. MainNet policy

MainNet must remain refused unless production authority criteria are satisfied.

Fixture/loopback/dev material must not satisfy MainNet.

Local operator material alone must not satisfy MainNet.

Peer majority must not satisfy MainNet.

The backend must not allow a RemoteSigner response alone to bypass governance proof verification, custody attestation, validator-set rotation requirements, or MainNet readiness gates.

7. No raw local production signing keys

The production RemoteSigner backend must not introduce raw local production signing-key loading.

Source/test loopback keys may exist only in tests/examples and must be clearly non-production.

No secrets, private keys, tokens, credentials, or cloud provider material may be committed.

8. Composition with existing authority/custody/governance surfaces

The backend should compose narrowly with existing Runs 188–202 authority custody / RemoteSigner boundary types.

Where existing validators already verify request/response/transcript material, reuse them rather than creating parallel semantics.

The backend must not bypass existing custody class, RemoteSigner policy, governance authority verifier, or MainNet peer-driven apply refusal.
