# RUN 098 — Wire Canonical `meta:current_epoch` into Production `ActivationContext.current_epoch`

## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

* Runs 050–068 completed signed PQC trust-bundle lifecycle.
* Runs 069–075 completed local operator reload lifecycle.
* Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
* Run 087 added peer-driven propagation/apply safety specification.
* Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
* Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
* Run 090 updated the operator runbook for propagation-only behavior.
* Run 091 established fail-closed `CurrentEpochUnavailable` behavior for `activation_epoch` whenever no canonical runtime epoch source exists.
* Run 092 proved production `qbind-node` did not open `RocksDbConsensusStorage`, so `meta:current_epoch` was phantom on real binary runs.
* Run 093 introduced canonical production binary-path `ConsensusStorage` ownership at `<data_dir>/consensus`.
* Run 094 threaded the Run 093 `ConsensusStorage` handle into the binary consensus loop and added fail-closed epoch persistence through existing atomic epoch persistence machinery.
* Run 095 added binary-path committed reconfig-block detection and engine epoch transition trigger.
* Run 096 added the smallest explicit local operator-gated DevNet/TestNet reconfig proposal source and proved the reconfig → transition → persistence path in tests.
* Run 097 added snapshot epoch parity:

  * `StateSnapshotMeta.epoch: Option<u64>`
  * old snapshots remain valid
  * missing epoch remains `None`, not epoch `0`
  * snapshot creation uses canonical `ConsensusStorage::get_current_epoch()`
  * restore can persist snapshot epoch back into `<data_dir>/consensus`
  * restore fails closed on epoch inconsistency/write failure
  * trust-bundle activation still remained isolated with `ActivationContext.current_epoch: None`

Run 098 is the first run allowed to consume the canonical epoch source for trust-bundle activation.

Current remaining C4 items include:

* activation_epoch runtime-source consumption
* peer-driven live apply
* KMS/HSM custody
* in-binary/on-chain signing-key ratification
* production fast-sync / broader consensus-storage restore
* per-environment production trust-anchor operation
* full C4/C5 closure

Run 098 must address only the first item:

> Wire canonical production `meta:current_epoch` into `ActivationContext.current_epoch` at all production trust-bundle activation call sites, while preserving fail-closed behavior whenever no committed epoch is available.

---

# Exact objective

Implement the smallest safe production wiring that supplies canonical committed epoch into trust-bundle activation checks.

The objective is:

1. Read `meta:current_epoch` only from the canonical Run 093 production `ConsensusStorage` surface.
2. Provide `Some(epoch)` to `ActivationContext.current_epoch` only when a committed epoch is actually present.
3. Preserve `CurrentEpochUnavailable` when:

   * fresh genesis has no committed epoch
   * old restored snapshot lacks epoch metadata
   * storage is unavailable
   * storage is corrupt
   * canonical epoch lookup fails
4. Apply the same behavior consistently across all production trust-bundle activation surfaces:

   * startup `--p2p-trust-bundle`
   * reload-check
   * reload-apply
   * SIGHUP live reload
   * peer-candidate validation
   * peer-candidate propagation
5. Preserve all existing activation-height, minimum-margin, revocation, sequence, and trust-bundle safety checks.
6. Do not invent or infer epoch from any non-canonical source.

Run 098 may finally narrow the C4 `activation_epoch runtime source` item, but only for canonical committed-epoch cases.

Run 098 must not claim full C4 or C5 closure.

---

# Strict scope

Allowed:

## A. Canonical epoch lookup helper

Add a narrow helper if needed, for example:

```rust
load_activation_current_epoch(...)
```

or equivalent.

It must:

* use only canonical production `ConsensusStorage`
* return `Some(epoch)` only when `meta:current_epoch` exists
* return `None` for explicit no-epoch state
* fail closed on storage corruption/read failure where appropriate
* never derive epoch from height/time/view/snapshot height

---

## B. ActivationContext wiring

Change production `ActivationContext` construction from:

```rust
current_epoch: None
```

to:

```rust
current_epoch: canonical_epoch_option
```

only where canonical epoch lookup is available and safe.

All production trust-bundle activation surfaces must be audited and updated consistently.

---

## C. Fail-closed no-epoch behavior

If a trust bundle or active root declares `activation_epoch` and canonical epoch is unavailable:

* behavior must remain `CurrentEpochUnavailable`
* no silent ignore
* no epoch `0` fallback
* no environment-specific bypass

If the bundle does not declare `activation_epoch`, existing behavior should remain unchanged.

---

## D. Fresh genesis and old snapshot behavior

Fresh genesis:

* no committed epoch
* `current_epoch = None`
* bundles requiring `activation_epoch` fail closed

Old snapshot without epoch:

* restored state may be valid
* no committed epoch restored
* `current_epoch = None`
* bundles requiring `activation_epoch` fail closed

---

## E. Committed/restored epoch behavior

If `meta:current_epoch = n` exists canonically:

* activation bundle/root with `activation_epoch <= n` may pass the epoch axis, subject to all other checks
* activation bundle/root with `activation_epoch > n` must fail closed as future epoch
* all existing height/minimum-margin/revocation/sequence checks still apply

---
