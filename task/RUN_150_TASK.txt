Objective:

Required design:
Add an explicit local trigger that drains at most one eligible staged candidate into the Run 148 controller.

The trigger must be:
- disabled by default;
- DevNet/TestNet only;
- MainNet refused;
- operator/local only;
- deterministic;
- bounded;
- observable in tests;
- unable to run concurrently with itself;
- unable to bypass validation-only, staging policy, v2 marker checks, Run 055 sequence anti-rollback, or Run 070 apply ordering.

Acceptable trigger forms:
Choose the narrowest existing-friendly option:
1. a hidden local CLI/process-start flag for test/dev evidence, or
2. an internal controller method exercised by tests and explicitly documented as the future binary hook, or
3. a Unix signal only if it does not collide with existing SIGHUP reload semantics.

Preferred for Run 150:
Use an internal explicit method first if it avoids new CLI surface. Defer release-binary operator trigger evidence to Run 151.

Required behavior:
1. Drain trigger checks policy first.
2. MainNet is always refused.
3. Disabled policy is refused.
4. Empty queue returns a typed no-op result.
5. Invalid/rejected candidates cannot be drained.
6. Expired staged candidates cannot be drained.
7. Duplicate staged candidates do not cause duplicate apply.
8. Drain selects exactly one candidate deterministically.
9. Drain calls the Run 148 controller, not Run 070 directly.
10. Run 148 controller calls the existing Run 070 apply contract.
11. Apply ordering remains exactly:
    validate → snapshot previous → swap LivePqcTrustState → evict sessions → commit_sequence
12. v2 authority marker persistence occurs only after successful sequence commit.
13. If validation/controller/apply fails before mutation, no mutation occurs.
14. If eviction or sequence commit fails, existing Run 070 rollback/fatal semantics are preserved.
15. Drain removes or marks the candidate only after a successful terminal result according to an explicit policy.
16. Drain must not silently drop failed candidates unless the failure class is permanently invalid.

Required result type:
Add a typed result enum similar to:

PeerDrivenDrainOutcome {
    Applied { candidate_fingerprint, sequence, authority_sequence, ... },
    NoCandidate,
    Disabled,
    MainNetRefused,
    CandidateExpired,
    CandidateRejectedBeforeApply(...),
    ApplyRejected(...),
    ApplyFatal(...),
    AlreadyInProgress,
}

Names can differ, but outcomes must be typed and testable.

Required concurrency guard:
Add an in-progress guard so two drains cannot apply two candidates concurrently.

Expected behavior:
- first trigger enters drain;
- concurrent trigger returns AlreadyInProgress;
- no double sequence write;
- no double marker write;
- no double session eviction;
- no double live trust swap.

Required candidate selection policy:
Define and test a deterministic selection rule, for example:
- highest authority-domain sequence wins;
- tie broken by candidate fingerprint;
- never select lower-sequence over higher-sequence;
- never select same-sequence conflicting digest;
- never select expired candidate.

If the queue already defines ordering, use it and document it.
