# Exact objective

Produce release-binary evidence proving that process-start reload-apply can safely consume v2 ratifications and persist v2 authority marker state only after the existing commit boundary.

Run 135 should prefer **no production runtime code changes**.

Allowed changes are limited to:

- release-binary evidence scripts;
- evidence-only fixture/helper generation;
- log/assertion helpers;
- documentation/evidence updates;
- narrow bug fixes if release-binary evidence exposes a real Run 134 defect.

The objective is to prove on real `target/release/qbind-node` binaries:

1. v2 no-marker reload-apply succeeds and persists a v2 marker after `commit_sequence`.
2. v1 marker → v2 migration reload-apply succeeds and persists a v2 marker after `commit_sequence`.
3. v2 same-sequence/same-digest idempotent reload-apply succeeds without unintended marker rewrite.
4. v2 higher-sequence reload-apply succeeds and persists updated v2 marker after `commit_sequence`, if fixture support is feasible.
5. lower-sequence v2 rejects before mutation.
6. same-sequence/different-digest v2 rejects before mutation.
7. bad v2 signature or wrong-domain v2 rejects before mutation.
8. apply failure before `commit_sequence` does not persist a v2 marker.
9. v1 reload-apply regression still passes.
10. Rejection cases do not alter marker bytes, sequence files, root state, live trust state, session state, or apply outputs.
11. No startup v2, SIGHUP v2, or live `0x05` v2 wiring is added.
12. No signing-key rotation/revocation lifecycle is implemented.

Run 135 should make this statement true:

> Real release binaries prove that v2 process-start reload-apply persists v2 authority marker state only after accepted mutation, refuses downgrade/replay/equivocation before mutation, and preserves v1 behavior.

# Required release-binary evidence scenarios

Produce logs, exit codes, marker before/after hashes, and summary.

## Scenario A1 — v2 no-marker first write succeeds

No local authority marker exists.

Run v2 reload-apply.

Prove:

- exit success;
- reload-apply commit occurred;
- v2 marker was persisted;
- marker is versioned V2;
- marker contains expected `authority_domain_sequence` and lifecycle action;
- no startup/SIGHUP/live `0x05` v2 markers appear.

## Scenario A2 — v1 marker → v2 migration succeeds

Seed v1 marker.

Run valid v2 reload-apply.

Prove:

- exit success;
- v1 marker is replaced by V2 marker only after commit;
- migration is explicit;
- no mutation occurs before marker preflight passes.

## Scenario A3 — idempotent same-sequence/same-digest succeeds

Seed matching V2 marker.

Run same v2 reload-apply.

Prove:

- exit success;
- marker bytes remain byte-identical if idempotent no-op is intended;
- no unintended rewrite.

## Scenario A4 — higher-sequence succeeds, if feasible

Seed lower V2 marker, run higher-sequence v2 reload-apply.

Prove:

- exit success;
- marker advances only after commit.

If not feasible, document source/test-only coverage from Run 134.

## Scenario R1 — lower-sequence rejects before mutation

Seed higher V2 marker, run lower-sequence v2 candidate.

Prove:

- non-zero exit;
- typed rejection log;
- marker bytes unchanged;
- no apply callbacks / sequence/root mutation.

## Scenario R2 — same-sequence/different-digest rejects before mutation

Seed V2 marker, run same sequence but different digest.

Prove same no-mutation invariants.

## Scenario R3 — bad-signature or wrong-domain rejects before mutation

Use one or more wrong-domain fixtures.

Prove no marker write or mutation.

## Scenario R4 — apply failure before commit does not persist marker

If feasible, trigger apply failure after marker preflight but before commit.

Prove:

- marker is not written/advanced;
- failure does not leave tmp marker file;
- existing marker unchanged.

If not feasible on release binary, document source/test-only coverage from Run 134.

## Scenario V1 — v1 reload-apply regression

Run an existing v1 reload-apply scenario.

Prove:

- v1 path still succeeds;
- v1 marker behavior remains unchanged;
- no v2 marker is written from v1 sidecar unless existing v1 behavior requires otherwise.
