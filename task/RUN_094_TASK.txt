# RUN 094 — Binary-Path Epoch Transition Persistence via Run 093 ConsensusStorage Handle

## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over convenience or speed.

Relevant recent state:

* Runs 050–068 completed signed PQC trust-bundle lifecycle.
* Runs 069–075 completed local operator reload lifecycle.
* Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
* Run 087 added peer-driven propagation/apply safety specification.
* Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
* Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
* Run 090 updated the operator runbook for propagation-only behavior.
* Run 091 established fail-closed `CurrentEpochUnavailable` behavior for `activation_epoch` whenever no canonical runtime epoch source exists.
* Run 092 proved that the production `qbind-node` binary did not open `RocksDbConsensusStorage`, so `meta:current_epoch` was phantom on real binary runs.
* Run 093 introduced canonical production binary-path `ConsensusStorage` ownership:

  * canonical path: `<data_dir>/consensus`
  * production storage open before network/consensus dispatch
  * schema compatibility checks
  * incomplete-transition/corruption guardrails
  * explicit state distinction:

    * `NoConsensusStorage`
    * `PresentNoCommittedEpoch`
    * `CommittedEpoch(u64)`
  * no trust-bundle activation consumption
  * Run 091/092 fail-closed behavior preserved

Run 093 was partial-positive because it established binary-path storage ownership but did **not** yet prove end-to-end release-binary committed-epoch persistence. The production consensus loop still does not write epoch transitions into `apply_epoch_transition_atomic`.

Current open C4 items include:

* binary-path epoch transition persistence
* activation_epoch real runtime source
* snapshot epoch parity
* peer-driven live apply
* KMS/HSM custody
* in-binary/on-chain signing-key ratification
* production fast-sync / consensus-storage restore
* per-environment production trust-anchor operation
* full C4/C5 closure

Run 094 must address only the first item: binary-path epoch transition persistence.

# Exact objective

Implement the smallest production-honest wiring that makes the real `qbind-node` binary persist committed epoch transitions through the Run 093 canonical `ConsensusStorage` handle.

The objective is:

1. Thread the Run 093 opened production `ConsensusStorage` handle into the binary consensus path.
2. Detect or receive real epoch transition events from the existing consensus/epoch machinery.
3. Persist committed epoch transitions using the existing atomic persistence surface, expected to be `apply_epoch_transition_atomic(...)` or the repo’s existing equivalent.
4. Prove on release binaries that `meta:current_epoch` is actually written and survives restart.
5. Preserve Run 091/092 fail-closed `CurrentEpochUnavailable` behavior for trust-bundle activation.

Run 094 must make epoch persistence real on the production binary path.

Run 094 must **not** consume `current_epoch` for trust-bundle activation.
