# Exact objective

Wire bundle-signing-key ratification enforcement into the process-start reload-apply path while preserving the Run 070 / Run 073 mutation ordering model.

The process-start reload-apply path is the binary path controlled by flags such as:

```text
--p2p-trust-bundle-reload-apply-enabled
--p2p-trust-bundle-reload-apply-path <PATH>
```

or the repo’s current equivalent.

The objective is:

1. Enforce bundle-signing-key ratification before any reload-apply mutation side effect.
2. Use the existing Run 103/104 verifier, Run 105 sidecar input model, and Run 106 per-environment policy.
3. Make MainNet/TestNet ratification mandatory by default.
4. Preserve DevNet explicit opt-in behavior if still required.
5. Reject missing, malformed, wrong-chain, wrong-environment, unknown-root, transport-root, missing-key-material, malformed-key-material, unsupported-suite, or bad-signature ratification before apply.
6. Preserve the existing Run 070 / Run 073 ordering:

```text
validate → snapshot → swap → evict_sessions → commit_sequence
```

7. Ensure ratification rejection happens before `snapshot`, `swap`, `evict_sessions`, and `commit_sequence`.
8. Preserve all existing trust-bundle checks:
   - signed-bundle verification;
   - chain_id;
   - environment;
   - Run 055 sequence anti-rollback;
   - activation-height;
   - activation_epoch;
   - Run 065 minimum-margin;
   - revocation activation;
   - local leaf self-check;
   - issuer-root self-check.
9. Do not wire SIGHUP live reload in this run.
10. Do not implement peer-driven live apply.

Run 112 should make this statement true:

> Process-start reload-apply cannot apply a trust bundle signed by an unratified bundle-signing key on MainNet/TestNet, and rejection occurs before any live trust mutation, session eviction, or sequence commit.
