# Exact objective

Produce release-binary evidence proving that process-start reload-apply enforces bundle-signing-key ratification before any mutation side effect.

Run 113 should prefer **no production runtime code changes**.

Allowed changes are limited to:

- DevNet evidence scripts;
- evidence-only fixture/helper generation;
- log/metrics assertion helpers;
- documentation/evidence updates;
- narrow bug fixes if release-binary evidence exposes a real Run 112 implementation defect.

The objective is to prove on real `target/release/qbind-node` binaries:

1. Valid ratification allows otherwise-valid process-start reload-apply to proceed.
2. Missing ratification rejects before mutation.
3. Bad ratification rejects before mutation.
4. Wrong-chain / wrong-environment / wrong-root ratification rejects before mutation if covered by the harness.
5. Rejection occurs before:
   - snapshot;
   - live trust-state swap;
   - session eviction;
   - sequence commit;
   - sequence-file write;
   - root merge.
6. Successful apply preserves the Run 070 / Run 073 ordering:

```text
validate → snapshot → swap → evict_sessions → commit_sequence
```

7. DevNet policy remains explicit:
   - no-opt-in behavior preserved if still intentionally allowed;
   - opt-in enforces ratification.
8. SIGHUP live reload remains untouched.
9. No peer-driven live apply is introduced.
