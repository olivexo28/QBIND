Implementation requirements:

1. Binary-reachable staging queue handle
- Ensure the live inbound 0x05 path and the drain-once hook can share the same staged-candidate queue instance.
- The queue must remain in-memory only.
- The queue must not persist staged candidates to disk.
- The queue must remain bounded, deduped, and disabled unless the existing staging/apply flags enable it.
- Existing validation-only and propagation-only behavior must remain unchanged.

2. Drain invocation builder
Implement or wire a production-capable builder that converts a staged candidate into the input expected by the Run 150 drain / Run 148 controller.

The builder must:
- consume only candidates already accepted by validation-only/staging;
- re-check candidate freshness/expiry;
- re-check environment / chain_id / genesis_hash / authority-root binding;
- re-check v2 marker relation before apply;
- produce deterministic candidate fingerprint / sequence / authority sequence metadata;
- fail closed on missing candidate material;
- fail closed on malformed staged metadata;
- fail closed on ambiguous v1+v2 material;
- never write marker or sequence files itself;
- never mutate `LivePqcTrustState` itself;
- never evict sessions itself;
- never call Run 070 directly.

3. V2 marker coordinator
Implement or wire a production-capable `V2MarkerCoordinator` for the peer-driven apply path.

It must:
- run the same v2 marker decision discipline used by Runs 134/136/138/150;
- perform no marker write before Run 070 sequence commit succeeds;
- persist v2 marker only after successful sequence commit;
- fail closed on lower sequence;
- fail closed on same-sequence different digest;
- fail closed on wrong domain;
- fail closed on corrupted local marker;
- surface marker-persist-after-commit failure as fatal/operator-actionable;
- preserve Run 151/150 MainNet refusal policy.

4. Hook integration
The hidden `--p2p-trust-bundle-peer-candidate-drain-once` path must:
- remain hidden;
- remain disabled by default;
- remain DevNet/TestNet-only;
- remain MainNet refused;
- require the existing apply/staging/wire-validation co-requisites;
- drain at most one candidate;
- use the shared staging queue;
- route through Run 150 drain and Run 148 controller;
- not call Run 070 directly;
- produce typed outcome logs.

5. Concurrency and lifecycle
- Preserve Run 150 `Arc<AtomicBool>` / RAII concurrency guard.
- No two drain invocations may run concurrently.
- Node must not autonomously re-run drain after success.
- Drain-once must be exactly one-shot.
- A second drain attempt after successful apply must return NoCandidate / AlreadyApplied / deduped result according to Run 150 policy.

6. Ordering
Accepted peer-driven apply must preserve exactly:

  validate → snapshot previous → swap LivePqcTrustState → evict_sessions → commit_sequence → persist v2 authority marker

The marker persist must be strictly after sequence commit.
