Implementation target:
Integrate lifecycle validation into the shared v2 marker-decision layer used by:

1. Run 134 process-start reload-apply v2 marker path.
2. Run 136 startup `--p2p-trust-bundle` v2 marker path.
3. Run 138 SIGHUP live-reload v2 marker path.
4. Run 150 peer-driven drain source/test path.
5. Run 152 production drain invocation / v2 marker coordinator path.

If reload-check / peer-candidate-check / live `0x05` validation-only paths also share the same v2 marker decision helper, they must receive lifecycle validation in validation-only mode without any disk writes.

Required design:
Add a narrow lifecycle-validation call inside the existing v2 marker comparison/decision helper, not as an independent parallel path.

The expected composition is:

1. Load existing persisted authority marker, if present.
2. Build `AuthorityTrustDomain` from the runtime environment, chain ID, genesis hash, and authority root.
3. Decode/classify the candidate lifecycle action from existing v2 ratification/action fields using `classify_local_lifecycle_action`.
4. Call `validate_v2_lifecycle_transition(persisted, candidate, trust_domain)`.
5. Combine the lifecycle outcome with the existing v2 marker anti-rollback decision.
6. Return one typed marker decision object.
7. Perform no disk write during preflight.
8. Persist marker only at existing post-commit points.

Required semantics:

* Existing v2 anti-rollback remains mandatory:

  * lower sequence rejected;
  * same-sequence different digest rejected;
  * same-sequence same digest idempotent;
  * wrong environment / chain / genesis / authority root rejected.
* Lifecycle validation adds:

  * wrong previous key rejected;
  * revoked key reuse rejected;
  * retired key reuse rejected;
  * malformed revoked metadata rejected;
  * non-PQC suite rejected;
  * unsupported lifecycle action rejected;
  * emergency revocation replay rejected.
* Accepted lifecycle actions:

  * ActivateInitial accepted only when no persisted marker or when existing rules allow first-write.
  * Rotate accepted only with higher sequence and correct previous key.
  * Retire accepted only with higher sequence and well-formed lifecycle metadata.
  * Revoke accepted only with higher sequence and well-formed revoked metadata.
  * EmergencyRevoke accepted only with higher sequence and valid emergency subclass metadata.
  * Idempotent same-record accepted as idempotent, not as a new mutation.
* Rejected lifecycle transitions must not:

  * write marker;
  * write sequence;
  * mutate `LivePqcTrustState`;
  * evict sessions;
  * call Run 070 apply;
  * allow propagation-to-apply.

Required ordering:
For all mutating surfaces:

* lifecycle validation must run before live mutation;
* marker persistence remains strictly after successful Run 055 sequence commit;
* if lifecycle validation fails, no Run 070 apply begins;
* if Run 070 apply fails before sequence commit, no marker persist;
* if marker persist fails after sequence commit, existing fatal/operator-actionable behavior remains.
