Task: Run 146 — Source/Test Wiring for Opt-In Live 0x05 Peer-Candidate Staging Hook

Current accepted state:
- Run 142/143: live inbound 0x05 v2 validation-only path is source/test wired and release-binary evidenced.
- Run 144: peer-driven live apply safety specification landed.
- Run 145: non-applying PeerCandidateStagingQueue scaffold landed as library-level source/test only.
- Run 145 intentionally does not wire staging into the live inbound 0x05 dispatcher.
- Therefore release-binary staging evidence is not yet honest until a runtime hook exists.

Objective:
Wire the Run 145 non-applying staging queue into the live inbound 0x05 validation-only path behind an explicit disabled-by-default local policy gate.

This run must allow a running node to stage already-validated peer candidates, while preserving the absolute invariant that staging does not apply, mutate live trust, write sequence files, write authority markers, evict sessions, or invoke Run 070.

This is source/test wiring only. Release-binary staging evidence is deferred to Run 147.

Strict scope:
- Source/test wiring only.
- No release-binary evidence harness in this run.
- No peer-driven live apply.
- No Run 070 apply invocation.
- No LivePqcTrustState mutation.
- No sequence write.
- No authority-marker write.
- No session eviction.
- No SIGHUP/reload-apply invocation.
- No new wire format.
- No trust-bundle schema change.
- No ratification sidecar schema change.
- No marker schema change.
- No KMS/HSM.
- No governance implementation.
- No signing-key rotation/revocation lifecycle.
- No MainNet staging enablement.
- Do not weaken validation-only or propagation-only behavior.
- Do not claim full C4 or C5 closure.

Implementation requirements:

1. Add an explicit staging runtime policy surface.

Prefer an internal config type that reuses Run 145 policy:

PeerDrivenStagingPolicy {
    enabled: bool,
    environment: Environment,
    allow_devnet: bool,
    allow_testnet: bool,
    allow_mainnet: bool,
    max_staged_candidates: usize,
    max_candidates_per_peer: usize,
    ttl_secs: u64,
}

Default must be disabled.

2. Add a runtime holder for the staging queue.

The live inbound 0x05 dispatcher may receive:

Option<Arc<Mutex<PeerCandidateStagingQueue>>>
or an equivalent concurrency-safe handle.

If the handle is None or policy disabled, behavior must remain exactly Run 143 validation-only / propagation-only.

3. Add a disabled-by-default local flag only if necessary.

If the binary needs a flag to arm the hook for future release-binary evidence, add a hidden DevNet/TestNet-only flag such as:

--p2p-trust-bundle-peer-candidate-staging-enabled

The flag must:
- be hidden from help if existing peer-candidate flags are hidden;
- default false;
- refuse MainNet unconditionally;
- require existing live 0x05 validation flag, if that is how the receiver path is armed;
- not imply propagation;
- not imply apply.

If a flag is not added in Run 146, document exactly how Run 147 can honestly trigger staging on the release binary. Do not leave Run 147 impossible.

4. Wire staging only after validation acceptance.

Flow must be:

receive 0x05
→ decode/size checks
→ existing validation-only path
→ v1/v2 ratification and marker validation-only checks
→ if accepted and local staging policy enabled: stage metadata
→ optional existing propagation-only handling, unchanged
→ no apply

Invalid candidates must never stage.

5. Stage only metadata, not authority.

The staged record must remain non-authoritative. It must not become:
- active trust state;
- persisted authority marker;
- persisted trust-bundle sequence;
- an implicit operator approval;
- a governance approval.

6. Preserve propagation-only semantics.

If propagation-only is enabled:
- valid candidates may rebroadcast only according to existing Run 088/143 rules;
- invalid candidates never rebroadcast;
- staging must not be a prerequisite for propagation unless explicitly documented;
- propagation must not imply staging unless policy is enabled.

7. Preserve MainNet refusal.

Even if a future flag is passed, MainNet must refuse staging until governance/ratification/KMS-HSM policy exists.
