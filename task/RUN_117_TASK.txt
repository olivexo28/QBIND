# Exact objective

Implement the smallest safe persistent authority anti-rollback marker and snapshot metadata extension for ratified bundle-signing authority state.

The objective is:

1. Add a durable authority-state marker type for ratified bundle-signing authority state.
2. Persist/load/compare that marker through a canonical, fail-closed storage surface.
3. Extend snapshot metadata to carry authority-state metadata without breaking old snapshots.
4. Define and test restore behavior so older authority state cannot silently roll back a node.
5. Keep the marker primitive separate from trust-bundle sequence, activation height, and activation epoch.
6. Do not wire the marker into every validation/apply surface yet unless it is extremely narrow and safe.
7. Do not implement signing-key rotation/revocation lifecycle yet.
8. Do not implement peer-driven live apply.

Run 117 should make this statement true at the storage/snapshot primitive level:

> QBIND can persist and compare ratified bundle-signing authority state, detect rollback/conflict/corruption, and carry authority-state metadata through snapshots without treating missing or stale state as safe on production paths.

---

# Critical design constraint

Current Run 103/104 ratification objects do not yet contain a monotonic per-key authority sequence/epoch suitable for signing-key rotation.

Therefore Run 117 must not pretend to solve rotation.

Allowed comparison basis for Run 117:

- version;
- chain_id;
- environment;
- genesis_hash;
- authority_root_fingerprint;
- ratified_bundle_signing_key_fingerprint;
- canonical_ratification_digest;
- optional authority_sequence/authority_epoch if already present;
- marker status.

If no monotonic field exists, the marker may protect against:

- wrong-chain rollback;
- wrong-environment rollback;
- wrong-genesis rollback;
- same-key/different-ratification conflict;
- authority root/key mismatch;
- stale snapshot conflict with existing local marker;
- corruption;

but it must not claim full rotation anti-rollback until a monotonic authority-sequence field exists.

# Strict scope

Allowed:

## A. Authority-state marker type

Add a narrow type, using repo naming style, such as:

```text
RatifiedAuthorityState
AuthorityAntiRollbackState
BundleSigningAuthorityMarker
```

Expected conceptual fields:

```text
version
chain_id
environment
genesis_hash
authority_root_fingerprint
ratified_bundle_signing_key_fingerprint
canonical_ratification_digest
authority_sequence or authority_epoch, if already available
status
```

Do not invent a fake monotonic sequence if one is not already canonical.

## B. Canonical serialization / digest

Implement deterministic serialization or reuse existing canonical encoding.

Requirements:

- stable across platforms;
- domain-separated;
- includes every security-relevant field;
- no JSON ordering ambiguity;
- no implicit defaults that silently change semantics.

Expected domain tag example:

```text
QBIND:AUTHORITY-STATE:v1
```

Follow repo conventions if different.

## C. Compare function

Implement a pure compare/validation function.

Expected outcomes:

- `AcceptFirstMarker` / equivalent;
- `IdempotentSameMarker`;
- `RejectWrongChain`;
- `RejectWrongEnvironment`;
- `RejectWrongGenesis`;
- `RejectAuthorityRootMismatch`;
- `RejectKeyMismatch`;
- `RejectSameStateDifferentDigest`;
- `RejectLowerAuthoritySequence` if a monotonic sequence exists;
- `RejectCorruptMarker`.

Return typed reasons, not boolean-only.

## D. Persistence API

Choose the narrowest durable location consistent with Run 116.

Preferred if appropriate:

- existing Run 093 production `ConsensusStorage` / RocksDB metadata;
- a clearly versioned key under existing production consensus storage.

Avoid ad hoc files if the existing durable schema is the correct place.

Expected APIs:

```text
load_authority_state(...)
compare_authority_state(...)
persist_authority_state_atomic(...)
```

or repo-style equivalents.

## E. Snapshot metadata extension

Extend snapshot metadata additively with optional authority-state metadata.

Requirements:

- old snapshots without authority state remain parse-compatible;
- missing authority metadata must not be treated as safe production authority state;
- new snapshots may include authority marker digest/state;
- restore must detect conflict between snapshot authority state and existing local persisted marker;
- snapshot restore must not silently roll back authority state.

Do not redesign snapshots broadly.

## F. Reset/recovery design

If implementing reset handling, it must be explicit and guarded.

The Run 116 model mentioned an `--allow-authority-state-reset` style recovery path. Run 117 may specify it but should avoid adding it unless necessary.

If added, it must be:

- impossible to use silently;
- environment-gated;
- logged/audited;
- release-evidenced;
- unavailable or extremely constrained on MainNet.

Prefer deferring reset CLI behavior unless required.
