Task:
Run 069. Define and implement the safest disabled-by-default trust-bundle hot-reload validation/staging boundary.

Primary objective:
Add a safe, non-mutating hot-reload validation/staging path that can validate a candidate trust bundle using the same security checks as startup, but does not yet apply live trust changes to active P2P state.

Important framing:
This is NOT full hot reload.
This is NOT peer-supplied/gossiped bundle acceptance.
This is NOT live root replacement.
This is NOT live session rekeying or session eviction.
This is NOT KMS/HSM.
This is NOT signing-key ratification.
This is NOT activation_epoch implementation.
This is NOT fast-sync restore.
Do not redesign KEMTLS.
Do not redesign consensus.
Do not weaken startup trust-bundle validation.
Do not silently fall back to --p2p-trusted-root.
Do not silently fall back to DummySig, DummyKem, or DummyAead.
Do not claim full C4 or C5 closure.

Strict scope:
- Implement a disabled-by-default hot-reload candidate validation/staging boundary.
- The candidate validation pipeline must reuse the same logic as startup as much as possible:
  - parse trust bundle
  - verify ML-DSA-44 bundle signature
  - validate environment
  - validate chain_id
  - validate root status/windows
  - validate bundle activation_height
  - validate Run 065 minimum activation-height policy
  - validate per-entry revocation activation_height
  - validate sequence monotonicity against the persisted sequence record
  - check local leaf fingerprint revocation
  - check local issuer-root revocation
- Do not apply the candidate bundle to live P2P trust state.
- Do not mutate active roots, active revocations, peer sessions, or existing KEMTLS sessions.
- Do not persist sequence for staged-only candidates unless the design explicitly says “validation-only, no persistence.”
- Recommended safest behavior for Run 069:
  - validation-only / dry-run mode does NOT update sequence persistence;
  - staging records candidate metadata separately from the accepted sequence state;
  - no live trust change is applied.
- If sequence persistence would be mutated by candidate validation, stop and redesign. A rejected or unapplied candidate must not burn sequence numbers.
- Crosscheck all code against existing design/spec. If contradictions or inconsistencies are found, update docs/whitepaper/contradiction.md.
