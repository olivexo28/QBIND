# Exact objective

Implement an offline-only authority-state reset CLI skeleton that performs strict validation, emits an audit record, and refuses unsafe cases by default.

The objective is:

1. Add a local/offline `authority-state-reset` command surface or repo-equivalent command.
2. Require explicit operator inputs; never run implicitly.
3. Refuse MainNet by default unless future governance artifact support exists.
4. Allow DevNet reset only under explicit operator command and full audit output.
5. Allow TestNet reset only under stricter explicit ceremony inputs and full audit output if the Run 126 policy permits.
6. Verify genesis hash, chain_id, environment, genesis authority config, ratification sidecar, authority root, ratified signing key, and target marker derivation.
7. Archive the old marker bytes/hash before writing any new marker.
8. Write an audit record for both success and refusal where feasible.
9. Persist a new authority marker only after all validation succeeds and only on allowed environments.
10. Refuse peer-triggered, startup-triggered, automatic, or local-config-only reset behavior.
11. Do not implement MainNet governance artifact verification yet.
12. Do not implement signing-key rotation/revocation lifecycle.
13. Do not invent a fake per-key monotonic authority sequence.

Run 127 should make this statement true:

> QBIND has an explicit offline reset command skeleton that can safely reset authority marker state only under tightly validated DevNet/TestNet recovery conditions, while refusing MainNet local reset and all implicit/peer-triggered bypasses.

# Strict scope

Allowed:

## A. Offline reset command skeleton

Add a command surface such as:

```text
qbind-node authority-state-reset ...
```

or the repo’s preferred equivalent.

The command must be offline/local only. It must not start networking, consensus, P2P, SIGHUP handlers, reload tasks, peer-candidate dispatchers, or normal node runtime.

## B. Required inputs

The command should require explicit inputs similar to:

```text
--data-dir <PATH>
--genesis-path <PATH>
--expected-genesis-hash <HASH>
--p2p-trust-bundle <BUNDLE>
--p2p-trust-bundle-ratification <RATIFICATION>
--output-audit <PATH>
--environment <devnet|testnet|mainnet>
--operator-note <TEXT or PATH>   # optional if repo style prefers
```

Use repo naming/style where possible.

Do not add a normal-startup flag like `--allow-authority-state-reset` for this run.

## C. Environment policy

Implement Run 126 policy:

### DevNet

May allow reset with explicit command, valid inputs, and audit record.

### TestNet

May allow reset if Run 126 policy permits, but must require explicit command, valid inputs, stricter refusal checks, and audit record.

### MainNet

Must refuse local reset by default.

A future governance/recovery artifact may be designed later, but Run 127 must not implement or fake it.

## D. Validation pipeline

The reset command must validate:

- genesis file exists and parses;
- canonical genesis hash matches `--expected-genesis-hash`;
- runtime environment matches genesis authority policy;
- chain_id matches expected environment/domain;
- genesis authority config is present and valid;
- trust bundle is signed and valid under existing checks where applicable;
- ratification sidecar is present and verifies under genesis-bound bundle-signing authority roots;
- transport roots cannot authorize bundle-signing keys;
- derived target authority marker is domain-consistent;
- existing marker is loaded, archived, and compared;
- corrupt existing marker is treated as a recovery case requiring audit, not silently repaired;
- no local-config-only MainNet authority is accepted.

## E. Audit record

Implement a deterministic audit record file, likely JSON if repo conventions allow.

It must include at least:

```text
record_version
action = authority_state_reset
environment
chain_id
genesis_hash
old_marker_present
old_marker_hash
old_marker_record_if_parseable
old_marker_raw_sha256
new_marker_hash
new_marker_record
ratification_hash
trust_bundle_fingerprint
snapshot_metadata_hash_if_any
operator_note_hash_or_value
binary_sha256_or_unavailable
binary_build_id_or_unavailable
result = success | refused
refusal_reason_if_any
timestamp_or_explicitly_no_wall_clock_if deterministic policy requires
```

If project policy avoids wall-clock in deterministic artifacts, document and use deterministic fields only. Do not use wall-clock for consensus-critical behavior.

## F. Typed refusal cases

Add typed refusal reasons, not boolean-only failures.

Expected refusal cases:

- MissingDataDir;
- MissingGenesisPath;
- MissingExpectedGenesisHash;
- GenesisHashMismatch;
- MissingAuthorityConfig;
- WrongEnvironment;
- WrongChainId;
- MainNetLocalResetUnsupported;

- MissingRatification;
- InvalidRatification;

- AuthorityKeyMaterialUnavailable;
- AuthorityKeyMaterialMalformed;
- TargetMarkerDerivationFailed;
- ExistingMarkerCorrupt;
- AuditOutputMissing;
- AuditWriteFailed;
- MarkerPersistFailed;
- PeerTriggeredResetUnsupported / RuntimeResetUnsupported if applicable.

Use repo-appropriate names.

## G. Marker write behavior

Only after all validation succeeds and the environment policy allows reset:

- write audit record or prepare audit record first according to safe ordering;
- persist target marker atomically using Run 117 persistence;
- preserve old marker archive/hash in audit output;
- never delete marker without replacement;
- never synthesize marker from legacy snapshot;
- never write marker from validation-only path.
