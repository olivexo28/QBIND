# Exact objective

Wire authority marker conflict enforcement into snapshot restore so a restore cannot silently roll back or conflict with the locally persisted ratified bundle-signing authority marker.

The objective is:

1. Consume Run 117 `AuthorityStateSnapshotMeta` during snapshot restore.
2. Compare snapshot authority marker metadata against the local persisted authority marker before accepting restore.
3. Reject conflicting, corrupt, wrong-domain, or stale authority marker restore states fail-closed.
4. Preserve legacy snapshot compatibility without inventing synthetic authority marker state.
5. Preserve Run 097 snapshot epoch parity behavior.
6. Do not implement reset/recovery flag in this run unless strictly necessary and explicitly scoped.
7. Do not invent a fake monotonic authority sequence.
8. Do not implement signing-key rotation/revocation lifecycle.
9. Do not implement peer-driven live apply.

Run 124 should make this statement true:

> Snapshot restore cannot silently downgrade, conflict with, or erase the locally persisted ratified bundle-signing authority marker; conflicting snapshot authority metadata fails closed before restore acceptance.

# Strict scope

Allowed:

## A. Restore-time authority marker comparison

At snapshot restore time:

- read snapshot metadata;
- extract `AuthorityStateSnapshotMeta`, if present;
- load local persisted authority marker, if present;
- compare snapshot marker metadata to local persisted marker;
- reject conflicts before accepting restore.

## B. Legacy snapshot compatibility

Preserve compatibility with old snapshots that lack authority metadata.

Allowed behavior must be explicit and environment-aware:

- DevNet/TestNet may accept legacy snapshots if no local authority marker exists, depending on existing policy;
- MainNet should fail closed or require explicit future recovery procedure when local marker exists and snapshot metadata is missing;
- do not infer or synthesize authority marker state from a legacy snapshot.

## C. Conflict/fail-closed behavior

Reject restore if:

- local marker is corrupt;
- snapshot marker metadata is malformed;
- snapshot marker conflicts with local marker;
- snapshot marker has wrong chain_id;
- snapshot marker has wrong environment;
- snapshot marker has wrong genesis_hash;
- snapshot marker would erase a local persisted marker;
- snapshot marker uses unsupported record version.

## D. Snapshot metadata creation/checking

Ensure snapshots created after Run 117/124 include authority marker metadata when a local marker exists.

If this was already implemented by Run 117, verify it and add restore-side tests.

Do not redesign snapshot metadata broadly.

## E. Restore path only

Run 124 is about snapshot restore conflict enforcement.

Do not change mutating trust-bundle surfaces except for regression preservation.
Do not change validation-only surfaces except for regression preservation.
