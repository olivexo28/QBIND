# RUN 104 — Genesis-Bound Authority Key Material Registry for Bundle-Signing Ratification

## Current state

QBIND is a post-quantum-secure L1 blockchain. Security, correctness, production safety, and evidence discipline take priority over speed or convenience.

Recent state:

- Runs 050–068 completed signed PQC trust-bundle lifecycle.
- Runs 069–075 completed local operator reload lifecycle.
- Runs 076–086 completed peer-candidate `0x05` validation-only lifecycle.
- Run 087 added peer-driven propagation/apply safety specification.
- Run 088 implemented disabled-by-default validation-before-rebroadcast propagation prototype.
- Run 089 produced strongest-positive N=3 DevNet release-binary propagation evidence.
- Run 090 updated the operator runbook for propagation-only behavior.
- Runs 091–099 established, wired, and partially release-binary-evidenced canonical `activation_epoch` runtime-source behavior while preserving fail-closed behavior when no committed epoch exists.
- Run 100 added the formal production trust-anchor authority and bundle-signing-key ratification model:
  - production authority must be genesis-bound;
  - production static source-code anchors are rejected;
  - MainNet local/dev shortcuts are rejected;
  - bundle-signing-key ratification is required before peer-driven apply;
  - peer-driven live apply remains forbidden until authority ratification, anti-rollback, custody, emergency controls, and release-binary evidence exist.
- Run 101 added additive genesis authority fields and canonical genesis verification API:
  - `GenesisConfig.authority: Option<GenesisAuthorityConfig>`;
  - structurally separate PQC transport roots and bundle-signing authority roots;
  - canonical genesis hash with domain separation;
  - environment-specific boot verification API;
  - no source-code MainNet root anchors;
  - no transport-root reuse as bundle-signing authority.
- Run 102 wired canonical genesis verification into the release binary startup path and fixed `--print-genesis-hash`:
  - release binary verifies canonical genesis authority before restore / metrics / network / consensus startup;
  - `--print-genesis-hash` prints the canonical parsed-genesis hash;
  - MainNet fails closed on missing/malformed/mismatched expected hash and missing/malformed authority;
  - DevNet embedded-genesis path remains explicit non-production behavior.
- Run 103 implemented the minimal bundle-signing-key ratification verifier primitive:
  - `crates/qbind-ledger/src/bundle_signing_ratification.rs`;
  - `BundleSigningRatification` schema;
  - deterministic length-prefixed canonical preimage;
  - domain separator `QBIND:BUNDLE-SIGNING-RATIFICATION:v1`;
  - typed `RatificationFailure` reasons;
  - PQC verification through existing `MlDsa44SignatureSuite` / suite `100`;
  - authority lookup restricted to `bundle_signing_authority_roots`;
  - explicit rejection of `pqc_transport_roots` as bundle-signing authority;
  - honest `AuthorityKeyMaterialUnavailable` fail-closed boundary when genesis stores only a short fingerprint;
  - no CLI flag, admin API, network listener, filesystem watcher, gossip surface, metric, dependency, or trust-bundle apply-path consumption.

Run 103 was positive for the verifier implementation but left the expected key-material boundary:

> If genesis stores only a short fingerprint for a bundle-signing authority root, the verifier cannot verify an ML-DSA-44 signature. It must fail closed with `AuthorityKeyMaterialUnavailable`.

Run 104 must address only that boundary.

The goal is to make genesis-bound bundle-signing authority roots carry enough validated ML-DSA-44 public key material for the Run 103 verifier to verify normal ratification signatures without overloading or misusing the short fingerprint field.

Run 104 must not wire ratification enforcement into trust-bundle apply paths yet. That is Run 105+.

---

# Exact objective

Implement a clean genesis-bound authority key material registry for bundle-signing ratification.

The objective is:

1. Extend genesis authority data so `bundle_signing_authority_roots` can carry full ML-DSA-44 public key material, not only short fingerprints.
2. Keep short fingerprints and full public key material structurally distinct.
3. Validate suite ID, public key length, fingerprint derivation, chain binding, environment binding, and duplicate/ambiguity rules.
4. Include the full authority key material in the canonical genesis hash from Run 101.
5. Preserve backward compatibility where appropriate, but fail closed for MainNet when full authority key material is missing.
6. Update Run 103 verifier inputs so a valid genesis authority root with key material can verify a real ratification signature.
7. Preserve `AuthorityKeyMaterialUnavailable` for legacy / incomplete authority roots instead of faking verification.
8. Do not consume ratification in trust-bundle validation/apply surfaces yet.

Run 104 should make this statement true at the library/verifier level:

> A genesis-bound bundle-signing authority root contains validated ML-DSA-44 public key material, and the Run 103 verifier can use it to verify a bundle-signing-key ratification signature.

# Strict scope

Allowed:

## A. Additive authority key material representation

Extend the genesis authority root model with a clear key-material representation.

Expected conceptual shape:

```text
genesis.authority.bundle_signing_authority_roots[]:
  root_id / fingerprint
  suite_id
  public_key_bytes
  key_fingerprint
  chain_id / chain binding if represented
  environment / environment binding if represented
  status / policy metadata if already present
```

Use project-appropriate names and types.

Requirements:

- full public key material must be distinct from short fingerprint;
- no overloading `key_fingerprint` as public key bytes;
- no transport-root reuse;
- no production static source-code keys;
- no hidden fallback key material.

## B. Fingerprint derivation and consistency

Implement or reuse a canonical fingerprint derivation for authority public keys.

Requirements:

- deterministic;
- domain-separated if project style requires it;
- suite-aware;
- public-key-length-aware;
- test-covered;
- mismatch between declared fingerprint and derived fingerprint must fail closed.

## C. Genesis canonical hash inclusion

Ensure full authority public key material affects canonical genesis hash.

Tests must prove:

- changing one byte of bundle-signing authority public key changes genesis hash;
- changing suite ID changes genesis hash;
- changing fingerprint changes genesis hash;
- changing transport roots does not get confused with bundle-signing authority roots.

## D. Environment-specific validation

MainNet must require complete key material for bundle-signing authority roots.

DevNet/TestNet may allow legacy short-fingerprint-only roots if existing tests/workflows require it, but this must be explicit and must preserve `AuthorityKeyMaterialUnavailable` in the verifier.

No missing key material may be silently accepted as verifiable authority.

## E. Run 103 verifier integration

Update the Run 103 verifier so that:

- when full authority public key material is present and valid, it verifies ratification signatures normally;
- when key material is absent, it still fails closed with `AuthorityKeyMaterialUnavailable`;
- when key material is malformed, it fails closed with a typed reason;
- when fingerprint/public key mismatch occurs, it fails closed;
- transport roots remain rejected.

## F. Evidence-only helper support if needed

If existing test/helper tooling cannot generate genesis authority roots with full key material, add narrow evidence/test helper support.

Allowed helper changes:

- DevNet/test genesis generator support;
- example/helper for producing ML-DSA-44 authority public key entries;
- test fixture generation.

Do not add runtime authority shortcuts.
