## What this run proves

1. legacy/no-attestation payloads remain compatible under default `Disabled`
   behavior;
2. fixture custody attestation reaches production-context paths in release
   mode where policy allows;
3. production / cloud-KMS / PKCS#11 / HSM / RemoteSigner attestation material
   reaches the verifier and fails closed as unavailable;
4. malformed/invalid custody-attestation material fails closed;
5. attestation evidence / input / transcript / provider-identity digests
   remain deterministic and domain-bound through wire conversion;
6. rejected cases produce no mutation;
7. MainNet peer-driven apply remains refused even with fixture attestation;
8. no real KMS / HSM attestation / RemoteSigner backend / governance
   execution / validator-set rotation is claimed.

## Honest limitations

* No real cloud KMS / PKCS#11 / HSM vendor attestation verifier is
  implemented. Carried production-class attestation material always routes
  into the Run 205 verifier and returns the typed unavailable outcome.
* Fixture custody attestation remains DevNet/TestNet evidence-only and is
  refused on a MainNet trust domain.
* No real RemoteSigner backend / networked signer daemon is implemented;
  RemoteSigner and KMS/HSM remain backend-boundary only.
* The `custody_attestation` sibling is additive and optional; legacy
  no-attestation payloads remain compatible and there is no schema / wire /
  metric drift beyond Run 207's additive optional sibling.
* governance execution remains unimplemented; real on-chain proof
  verification remains unimplemented; validator-set rotation remains open.
* MainNet peer-driven apply remains refused (Run 147 FATAL invariant) even
  with fixture attestation material.
* Full C4 remains open. C5 remains open.
