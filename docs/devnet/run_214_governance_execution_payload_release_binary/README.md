# Run 214 — Release-binary governance execution payload/carrying evidence

## Scope

Closes the Run 213-deferred release-binary boundary for the source/test
governance **execution payload / carrying** and production-context call-site
wiring added by
[`crates/qbind-node/src/pqc_governance_execution_payload_carrying.rs`](
  ../../../crates/qbind-node/src/pqc_governance_execution_payload_carrying.rs).
Run 213 added, at the source/test level only, the wire/carrying surface that
lets a v2 ratification sidecar carry governance execution input/decision
material into the Run 211 governance execution evaluator:

* the typed [`GovernanceExecutionClassWire`], [`GovernanceExecutionActionWire`],
  [`GovernanceExecutionInputWire`], [`GovernanceExecutionDecisionWire`], and
  [`GovernanceExecutionPayloadWire`] wire types, plus the
  [`GovernanceExecutionParts`] reconstruction, which convert into and out of the
  Run 211 internal governance execution input/decision material while preserving
  the deterministic input/decision/transcript/policy digests;
* the additive optional `governance_execution` JSON sibling on the v2
  ratification sidecar
  ([`GOVERNANCE_EXECUTION_PAYLOAD_SIBLING_FIELD`],
  [`GOVERNANCE_EXECUTION_PAYLOAD_WIRE_SCHEMA_VERSION`]), parsed by
  [`parse_optional_governance_execution_sibling_from_json_value`] and the
  `load_v2_ratification_sidecar_with_governance_execution_from_{bytes,path}`
  loaders into the typed [`GovernanceExecutionLoadStatus`]
  (`Absent` / `Available` / `Malformed`);
* the seven production-context routing helpers
  (`route_loaded_governance_execution_to_*_callsite_decision` for reload-check,
  reload-apply, startup `--p2p-trust-bundle`, SIGHUP, local peer-candidate-check,
  live inbound `0x05`, and peer-driven drain), the
  [`callsite_context_for_governance_execution`] constructor, and the
  [`GovernanceExecutionPayloadCarryingDecisionOutcome`];
* the [`evaluate_loaded_governance_execution`] and
  [`evaluate_loaded_governance_execution_with_peer_driven_guard`] entry points
  into the Run 211 [`evaluate_governance_execution_policy`] evaluator, plus
  [`mainnet_peer_driven_apply_remains_refused_under_governance_execution_payload_carrying`].

Run 213 is source/test only with the corpus
[`crates/qbind-node/tests/run_213_governance_execution_payload_callsite_tests.rs`](
  ../../../crates/qbind-node/tests/run_213_governance_execution_payload_callsite_tests.rs).
**Run 214 is the release-binary evidence run** for that boundary: it proves the
governance-execution payload/carrying boundary on the **real**
`target/release/qbind-node` plus a release-built helper linking the production
library symbols.

## What this run proves

1. legacy/no-governance-execution payloads remain compatible under default
   Disabled behavior;
2. fixture governance execution material reaches production-context paths in
   release mode where policy allows (DevNet/TestNet evidence-only);
3. production/on-chain/MainNet governance execution material reaches the
   evaluator and fails closed as unavailable;
4. malformed/invalid governance execution material fails closed;
5. governance execution input/decision/transcript/policy digests remain
   deterministic and domain-bound through wire conversion;
6. a carried governance decision authorizes a lifecycle action only when the
   action, candidate digest, and sequence match;
7. validator-set rotation remains unsupported;
8. rejected governance-execution-payload cases produce no mutation;
9. MainNet peer-driven apply remains refused even with fixture governance
   approval;
10. no real governance execution engine, on-chain proof verifier, KMS/HSM
    backend, RemoteSigner backend, or validator-set rotation is claimed.

## Real-binary surface

* `target/release/qbind-node --help` exposes no governance-execution surface and
  advertises no production / MainNet governance enablement and no on-chain
  governance proof verifier;
* default behaviour (no CLI, no env) does not expose or enable governance
  execution payload carrying;
* every existing Run 070 / 130–213 surface
  (`--print-genesis-hash --env devnet|testnet|mainnet`, the Run 193 custody
  selector, the Run 198 RemoteSigner selector, the Run 209 custody-attestation
  selector, the governance on-chain fixture flag) stays governance-execution
  silent and preserves the MainNet peer-driven apply refusal even on MainNet.

## Reproduce

```bash
bash scripts/devnet/run_214_governance_execution_payload_release_binary.sh
```

The harness is idempotent: it wipes and regenerates everything under this
directory except `README.md`, `summary.txt`, and `.gitignore`.

## Honest limitations

* No real governance execution engine is implemented. Production / on-chain /
  MainNet governance execution always returns the typed unavailable outcome,
  regardless of inputs.
* No real on-chain governance proof verifier is implemented.
* Fixture governance execution remains DevNet/TestNet evidence-only and is
  refused on a MainNet trust domain.
* Emergency council fixture is explicit and non-production.
* The Run 213 boundary is consumed at the library level; the release binary
  does not yet wire a governance execution evaluator into a long-running node
  runtime — no production governance is enabled.
* No real KMS / HSM backend, no real RemoteSigner backend, and no production
  signing-key custody; KMS/HSM/RemoteSigner/custody-attestation remain
  boundary-only and unchanged. There is no schema / wire / metric drift beyond
  Run 213's additive optional governance-execution sibling.
* Validator-set rotation remains unsupported.
* MainNet peer-driven apply remains refused (Run 147 FATAL invariant) even with
  a fixture governance approval.
* Full C4 remains open. C5 remains open.
