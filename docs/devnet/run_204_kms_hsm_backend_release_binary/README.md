# Run 204 — Release-binary KMS/HSM backend abstraction boundary evidence

## Scope

Closes the Run 203-deferred release-binary boundary for the source/test
production **KMS/HSM custody backend abstraction boundary** added by
[`crates/qbind-node/src/pqc_authority_kms_hsm_backend.rs`](
  ../../../crates/qbind-node/src/pqc_authority_kms_hsm_backend.rs).
Run 203 added the typed backend abstraction on top of the Run 188 typed
authority-custody boundary
[`crates/qbind-node/src/pqc_authority_custody.rs`](
  ../../../crates/qbind-node/src/pqc_authority_custody.rs):

* the backend kind / policy taxonomy `BackendKind` / `BackendPolicy`
  (disabled, fixture KMS, fixture HSM, production-KMS-required,
  production-HSM-required, mainnet-production-custody-required, and the
  cloud-KMS / PKCS#11 / production unavailable kinds);
* the `BackendIdentity` / `BackendRequest` / `BackendResponse` structs
  (authority root / signing-key fingerprints, environment / chain /
  genesis, custody key id, provider/backend id, `suite_id`, nonces,
  freshness/expiry, attestation/signature sentinels);
* deterministic, domain-separated identity / request / response digests
  (`identity_digest`, `request_digest`, `response_digest`) and the
  `backend_transcript_digest` binding;
* the pure/mockable `AuthorityCustodyBackend` trait with the
  DevNet/TestNet source/test-only `FixtureKmsBackend` /
  `FixtureHsmBackend` and the fail-closed `ProductionKmsBackend` /
  `ProductionHsmBackend` / `CloudKmsBackend` / `Pkcs11HsmBackend`;
* the typed `BackendOutcome` taxonomy;
* the verifier `verify_authority_custody_backend_response`, the
  custody-class router `validate_backend_for_custody_class`, and the
  composition `validate_lifecycle_governance_custody_and_backend`.

Run 203 is source/test only with the A1–A15 / R1–R41 corpus
[`crates/qbind-node/tests/run_203_kms_hsm_backend_boundary_tests.rs`](
  ../../../crates/qbind-node/tests/run_203_kms_hsm_backend_boundary_tests.rs).
**Run 204 is the release-binary evidence run** for that surface: it
proves the backend boundary on the **real** `target/release/qbind-node`
plus a release-built helper linking the production library symbols.

## What this run proves

1. fixture KMS/HSM backends remain DevNet/TestNet evidence-only;
2. production KMS/HSM backends remain unavailable / fail-closed;
3. cloud KMS and PKCS#11 HSM remain unavailable / fail-closed;
4. identity, request, response, and transcript digests are deterministic
   and domain-bound;
5. the backend boundary composes with the Run 188 custody classes;
6. the backend boundary composes with the lifecycle / governance /
   custody validation path at release-helper / library level;
7. MainNet peer-driven apply remains refused;
8. the RemoteSigner path remains separate and unchanged;
9. no real KMS / HSM / cloud KMS / PKCS#11 / RemoteSigner backend /
   governance execution / validator-set rotation is claimed.

## Reproduce

```bash
bash scripts/devnet/run_204_kms_hsm_backend_release_binary.sh
```

The harness is idempotent: it wipes and regenerates everything under this
directory except `README.md`, `summary.txt`, and `.gitignore`.

## Honest limitations

* No real KMS / HSM / cloud KMS / PKCS#11 backend is implemented. The
  `ProductionKmsBackend` / `ProductionHsmBackend` / `CloudKmsBackend` /
  `Pkcs11HsmBackend` always fail closed as `ProductionKmsUnavailable` /
  `ProductionHsmUnavailable` / `CloudKmsUnavailable` /
  `Pkcs11HsmUnavailable`.
* Fixture KMS/HSM backends remain DevNet/TestNet evidence-only and are
  refused on a MainNet trust domain.
* No real RemoteSigner backend / networked signer daemon is implemented;
  the RemoteSigner path remains separate and unchanged.
* governance execution remains unimplemented; real on-chain proof
  verification remains unimplemented; validator-set rotation remains open.
* MainNet peer-driven apply remains refused (Run 147 FATAL invariant).
* Full C4 remains open. C5 remains open.
