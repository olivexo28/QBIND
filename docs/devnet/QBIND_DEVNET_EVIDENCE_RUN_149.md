# QBIND DevNet Evidence — Run 149

**Subject:** Release-binary feasibility and minimal binary arming for peer-driven DevNet/TestNet trust-bundle apply through the existing Run 148 controller and Run 070 apply contract.

## Verdict

Run 149 is **minimal source wiring + release-binary evidence**, not pure evidence-only.

Feasibility gate answer: **No**, the pre-Run-149 `target/release/qbind-node` could not arm or invoke the Run 148 peer-driven apply controller through an existing runtime path. Run 148 exposed `try_apply_staged_peer_candidate` only as a library/test surface; Run 147 installed a non-applying staging queue only.

## Source delta

Run 149 adds the smallest hidden arming surface:

- `--p2p-trust-bundle-peer-candidate-apply-enabled`
  - hidden from help;
  - default false;
  - refused on MainNet unconditionally;
  - requires live inbound `0x05` validation;
  - requires peer-candidate staging;
  - does not imply propagation;
  - invokes the Run 148 controller only after validation, v2 marker validation-only checks, and staging.

The live dispatcher persists the validated frame bytes to an operator-controlled scratch path and calls `try_apply_staged_peer_candidate`, which delegates to Run 070 `apply_validated_candidate_with_previous`. The v2 marker coordinator persists only after Run 070 sequence commit.
